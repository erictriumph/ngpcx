# NGPCX — ARM Readiness Scanner

## What this is
Free Windows ARM (Snapdragon) readiness scanner. Consumer-facing brand: CheckMyARM — live on the visitor-facing pages (see Branding & monetization section); redirect domain still pending. Learning project with real utility; affiliate monetization is a bonus, not the goal.

## Stack — do not deviate without asking
- Server: Node.js + Express, **plain JavaScript, no TypeScript**. SQLite (better-sqlite3). Runs on Railway, auto-deploys from GitHub on push.
- Scanner: Rust .exe (`scanner/src/main.rs`), runs locally on the user's Windows machine only.
- No React, no Next.js, no Tailwind. No running the scanner from the web server. No running scrapers on Windows in production.

## Architecture essentials
- Scan level (quick/standard/full) travels through the localhost:7878 handshake between browser and scanner — never CLI-flag-only, never separate exe builds. CLI flags exist for dev convenience and always win over the browser-provided level (see `get_cli_scan_mode()` precedence in `main.rs`).
- **"Light/Standard/Advanced Assessment" is product-facing terminology layered over the same `quick`/`standard`/`full` wire values** — deliberately not a rename of the wire protocol, matching the existing NGPCX/CheckMyARM precedent (internal names stay stable, consumer-facing names can diverge). `results.scan_mode`, the `?level=` URL param, the scanner's CLI flags (`--quick`/`--standard`/`--full`), and `get_server_url()` all keep their original values; only display text in `index.html`/`report.html` changed. See "Assessment levels" below for what actually differs between them beyond naming.
- `--local` flag shortens the 7878 timeout to 3s (vs 20s prod) for faster dev iteration.
- Session IDs use `crypto.randomUUID()` (Node). Sessions expire after 24h — this is enforced server-side in both `/api/session/:id` and the scan-submission handler in `scan.js`, not just written and ignored.
- All HTML rendering of app/device names goes through an `escapeHtml()` helper (defined separately in each of `report.html`, `admin-research.html`, `admin-revisit.html` — no shared JS file) — never insert external data into `innerHTML` unescaped. This project has a real stored-XSS history (device FriendlyName is attacker-influenceable via USB descriptors); do not reintroduce raw interpolation. `escapeHtml()` encodes `"`/`'` in addition to `&`/`<`/`>` — needed because app/device names get interpolated into HTML *attributes* (`data-name="..."`, etc.) in the admin pages, not just text nodes, and a name containing a quote character could otherwise break out of the attribute. Found and fixed this gap once already; don't reintroduce a version that only handles text-node context.
- Readiness score (`classifyApps()` in `scan.js`) counts Unknown apps as **presumed emulated** (same 60-point weight as confirmed Emulated), not excluded — Windows on ARM runs ordinary x86/x64 apps under emulation by default, so true incompatibility is the rare case, not "no data either way." The separate confidence indicator (`report.html`, "High/Moderate/Limited Confidence — N of M apps matched") deliberately stays scoped to native+emulated+unsupported only, excluding Unknown — it answers "how much do we actually know," a different question from the score's "how likely is this to work." Don't collapse these two back into one number.

## Data sources for apps (in `server/scrapers/`)
- `winget.js` — GitHub-search-based, ARM64 installer detection, confidence 0.95
- `community/worksonwoa.js` — WorksOnWoA DB, confidence 0.75–0.90. Only maps `compatibility: 'no'` → `unsupported`; genuinely captures negative data when the source has it, but the source itself is ~99.9% "yes" (checked the raw feed: 6 "no" entries out of 8,772) — it's a "here's what works" community database, not a "here's what's broken" one. Don't assume there's untapped negative data being discarded here; there isn't.
- `github-lookup.js` — on-demand, single-app GitHub repo search + latest-release asset check (arm64/aarch64/win-arm64 in filenames), confidence 0.70. Triggered fire-and-forget from `scan.js` *after* the scan response is sent, for Unknown apps (cooldown via `unknown_apps.last_lookup_attempt`) and Emulated/Unsupported apps (cooldown via `apps.last_updated`) — never blocks the triggering user's report. Requires a strict normalized-name match on repo name (not `full_name`) plus a minimum star count (`MIN_STARS`, currently 20) before accepting a match — a wrong match is worse than no match. Confidence is deliberately kept below WorksOnWoA's floor (0.75) so it can never silently override a real source, only fill gaps or reinforce agreement with another github-auto entry.
- ARP (`get_arp_apps()` in `main.rs`) — Windows Uninstall registry, supplements winget for apps it misses (Office, Adobe, vendor tools). No confidence scoring; feeds `merge_arp_into_apps()` which dedupes against winget-found apps by normalized name — mode-agnostic (see App enumeration section). Reads `HKLM` (native + `Wow6432Node`) **and `HKCU:\...\Uninstall\*`** (Scanner Enrichment milestone, below) — no `HKCU\Wow6432Node` equivalent is needed, since WOW64 registry redirection only applies to `HKLM`, not per-user hives. HKCU coverage surfaces genuinely per-user installs invisible to HKLM: Chrome/Edge-installed PWAs (`Publisher` reads e.g. `Google\Chrome`), per-user dev-tool installs, and similar — confirmed on real hardware (several genuine Chrome PWAs, e.g. Google Calendar, TripIt, a stale ChatGPT PWA, previously invisible to every scan regardless of assessment level).
- **Never build a scraper against Windows Update Catalog or Microsoft Store's undocumented catalog API** — both investigated and rejected: Catalog requires POST+viewstate (a plain GET returns a static shell) and conflicts with Microsoft's ToU on reproducing catalog content; Store's catalog API is undocumented with unclear ToS. If asked to revisit either, flag the prior investigation before proceeding.

## Report refresh + on-demand lookups
- The classification loop lives in `classifyApps(apps)` (`scan.js`), extracted specifically so it can be re-run later against current DB state without a new physical scan. `POST /scan` calls it once at submission time; `sessions.raw_apps` persists the exact submitted app list so `POST /api/session/:id/refresh` (`server.js`) can call `classifyApps()` again later and pick up anything an admin resolved, a community submission accepted, or `github-lookup.js` found in the meantime.
- `report.html` shows a "Refresh results" banner whenever anything's still Unknown/Emulated/Unsupported, with copy that's deliberately honest about the mechanism: the automated check already ran once (fire-and-forget, right after the original scan), not an ongoing background process — repeatedly clicking refresh with no change is expected, not broken, since nothing new happens until the 14-day lookup cooldown elapses or a human (admin/community) acts.
- **Idempotency gotcha, don't reintroduce:** `renderReport()` can run more than once per page load (initial poll, then again on every refresh click). Any note it inserts into the DOM (`quick-mode-note`, `standard-mode-note`, `system-component-note`) must be removed before being re-added (`document.getElementById(id)?.remove()`), or it duplicates on every refresh — this was a real bug found via actual use, not just review.

## Device/driver detection (`main.rs`)
- Classes scanned: Printer, Image, MEDIA, Biometric, SmartCardReader, Camera, HIDClass, **Net, Display**. The last two were added specifically to catch USB-Ethernet/WiFi adapters and DisplayLink-style USB docks/graphics dongles — historically the worst ARM Windows compatibility pain point (Surface Pro X dock issues) — previously invisible to the scanner entirely. Filtered to `VID_xxxx&PID_xxxx` pattern InstanceIds (excludes `SW\` virtual bus devices); this same filter is the primary defense against virtual/software network adapters (VPN TAP, Hyper-V vswitches) showing up as Net-class noise, since those don't carry real USB VID/PID identifiers.
- Composite devices (one physical device, multiple USB interfaces) are deduped by VID/PID. Display name prefers non-generic interface names; "fido" is explicitly preserved even though it matches the generic-name filter (security keys must not display as "Keyboard Generic"). Classification uses the **most conservative class across all interfaces** — a device with any Biometric/SmartCardReader interface stays flagged for review even if it also exposes a generic HID interface.
- Recency signal is `IsPresent` OR `LastRemovalDate` OR `LastArrivalDate` (in that fallback order) — never arrival date alone. A continuously-connected device never updates arrival date; a recently-unplugged device is better signaled by removal date than by a stale arrival date. Tiers: Quick = present only (no per-device property lookups — this is what makes Quick fast); Full = everything. **Standard's window is per-class**, via `standard_threshold_days()`: 14 days for Camera/HIDClass/MEDIA/Biometric/SmartCardReader (these tend to cluster into "near-daily" or "effectively abandoned" — printers already bypass this filter entirely), 60 days for Net/Display (a travel dock or backup-drive adapter used monthly is a real pattern, closer to how apps are tiered than to a webcam).
- **Gotcha, don't reintroduce:** `Get-PnpDevice -PresentOnly:$false` means "show ONLY non-present devices," not "ignore presence." Omit the parameter entirely for true unfiltered enumeration.
- Printers via `Get-Printer` (not `Get-PnpDevice` — network printers have no USB VID/PID at all). Deduped by `DriverName`. Always included regardless of scan tier (no reliable recency signal exists for them).
- "Likely native" driver-check heuristic: device class is Camera or HIDClass (standard USB device classes with Windows-inbox drivers — UVC, HID) — validated against real hardware (webcam, BT-dongle-as-USB-audio, HID input), unconditionally "Likely native." **MEDIA is handled separately** (see Scanner Stabilization milestone below): it covers both plain USB Audio Class devices (mic/headset — genuinely fine) and professional audio interfaces (often vendor ASIO drivers, not fine) which the class alone can't distinguish, so it requires a `driver_provider` observation containing "Microsoft" before claiming native, falling back to "Check" without that evidence (e.g. Quick mode, where `driver_provider` isn't collected at all). Stays "worth a check" (search link, no claim) for Biometric/SmartCardReader and vendor-specific printer drivers — validated as genuinely ambiguous via real search results (Kensington fingerprint reader case).
- Driver/app "Check" links point to a plain web search (`"<name> ARM64 driver"` or similar) — **not** Windows Update Catalog (see rejection above).

## App enumeration is comprehensive and mode-agnostic (`main.rs`)
**Every mode returns the complete practical application inventory.** `get_installed_apps()` and `merge_arp_into_apps()` no longer take a `mode` parameter and no longer filter which discovered apps get pushed into the outgoing payload — `winget export` and the ARP registry read were already unconditional regardless of mode (confirmed by direct inspection: no mode-dependent enumeration cost exists for apps), so the old per-mode inclusion filter was only ever discarding apps *after* the expensive work was already done, for no real time savings. This used to be `include_by_recency(mode, is_running, days_ago)`, gating both the winget push site and `merge_arp_into_apps()` — deleted entirely, not deprecated, once both call sites stopped needing it. The "Quick mode found no apps (Prefetch unavailable) — falling back to standard" retry in `main()` is gone too, since that condition can no longer occur.
- `days_ago`/`is_running`/`recently_used` are still computed and attached to every app as enrichment (unchanged) — they just no longer decide inclusion. **The scanner observes and reports; `assessment.js` decides which applications are emphasized or counted toward a Light Assessment's recommendation** (see Assessment levels below). Don't reintroduce a mode-based app filter in `main.rs` — if a future need arises to reduce apps sent over the wire, that's a payload-size optimization, a different problem from what this section used to conflate it with.
- Devices are the only remaining place scan mode changes actual observation cost or scope — see Device/driver detection above (`get_devices(mode)`'s two genuinely different PowerShell paths, Quick present-only vs Standard/Full per-device property lookups). Measured directly: a Light-mode scan on a real machine completed in ~15s; the identical machine at Standard took ~86s, almost entirely spent in per-device `Get-PnpDeviceProperty` calls. Don't collapse this distinction back into the app-side one just deleted above — they're now deliberately different concerns with different owners (scanner owns device-observation depth; `assessment.js` owns application emphasis).

## Assessment levels (`assessment.js`, `report.html`, `workspace.html`)
Application population selection is now entirely the assessment layer's job, operating on the full inventory the scanner always reports (see above) — there is no scanner-side app filter left to conflict with it. "Light/Standard/Advanced Assessment" is product-facing terminology layered over the same `quick`/`standard`/`full` wire values (see Architecture essentials) — assessment level and scanner observation depth are related (the same `scan_mode` value drives both) but conceptually distinct: scan mode controls device-observation depth only; assessment level controls application emphasis, synthesis population, and Workspace complexity.

- **`isLightPriorityApp(app)`** (assessment.js): `is_running || is_startup || recently_used`. Deliberately excludes `has_start_menu_entry` — real-scan data (52 classified apps once app enumeration stopped being mode-filtered) showed it matches roughly half a typical machine's apps, which would make Light indistinguishable from Standard. `recently_used` is the older, coarser Prefetch-derived flag (pre-dates the Observation Enrichment Guidance Signals) — kept as a bonus signal, not load-bearing, since it's frequently empty on Prefetch-disabled machines. Verified against a real 52-app classified population: 11 apps qualify as Light-priority, 41 deprioritized-but-visible — a real, meaningfully curated result, not the ~7-8-total-app degenerate case from before this section's app-enumeration fix.
- **`isLightDeprioritized(app, decision, scanMode)`**: only ever `true` when `scanMode === 'quick'` — Standard and Advanced never deprioritize anything. A deprioritized app is excluded from `computeSynthesis()`'s weighted population entirely (same mechanism as `no_longer_use`, tracked separately as `deprioritizedCount`/`deprioritized`) — it still appears in its normal Native/Emulated/Unsupported/Unknown table on Results, unchanged; only the Score/Confidence/Primary Reasons math skips it.
- **Promotion is "This matters to me" setting `critical_to_me: true`** — not a new stored field or a second importance concept, the same flag `computeSynthesis()` already uses for weight-doubling and confidence-capping elsewhere. Once set, `isLightDeprioritized()` returns `false` for that item regardless of the heuristic. Devices are never deprioritized (a Light-scanned device list from `get_devices("quick")` is already just "currently connected," which is already Light's natural population) — "This device matters to me" is offered anyway, for symmetry and to future-proof a currently-inert flag, but has no computational effect today since devices don't feed the score in any mode.
- **Results**: a collapsed "Not Currently Prioritized" section (`renderNotPrioritized()`, only rendered when non-empty) lists every deprioritized app with a read-only Workspace deep-link — same pattern as System Components. Orthogonal to the pre-existing reduction-narrative ("Needs Attention" preview): that's about research status, this is about scoring inclusion; an app can be in both, neither, or either.
- **Workspace**: a 4th grouping tier, "Not Currently Prioritized" (`bucketItems()`/`renderGroups()`), inserted before Already Considered — only shown for the apps container, only under Light, only when non-empty. Cards in this group render minimally (name, evidence badge, research link, one promote button) with the full decision panel (findings, evidence detail, notes) still present underneath but gated behind Deeper Research Mode like everywhere else, not a new disclosure mechanism.
- **Store/AppX apps now participate in classification, not just as Guidance Signals — fixed in the Scanner Enrichment milestone (below).** This section previously documented WhatsApp/LinkedIn/ChatGPT Desktop as invisible to every classified bucket at every assessment level, since Store/AppX apps have no ARP/winget registration and `appx_apps` was deliberately never merged into `classifyApps()`. Superseded: `scan.js`'s `POST /scan` handler now merges `appx_apps` into the same population `classifyApps()` scores everything else against (`classifyPopulation = apps.concat(appxApps)`) — verified end-to-end on real hardware: WhatsApp now resolves `native`, LinkedIn resolves `x64-emulated`, ChatGPT Classic lands in `unknown` (no catalog entry — the honest outcome for an app the DB has no data on, not a bug). The scanner already guarantees `appx_apps` is disjoint from `apps` (deduped before the payload is sent), so no additional dedup was needed server-side. `sessions.raw_apps` now persists the merged population, not just the original `apps` array, so a later `/refresh` re-classification doesn't silently drop AppX apps back out. `appx_apps` is no longer passed through as a separate report field — redundant with the classified buckets now (confirmed nothing in any UI ever read `results.appx_apps` before removing it).
- **Accepted trade-off, not fixed:** merging `appx_apps` into `classifyApps()` means AppX component-shim noise — already documented as accepted residual noise in the Observation Enrichment milestone below, packages `get_appx_apps()`'s `IsFramework`/`SignatureKind` filtering can't catch — now also calls `trackUnknownApp()`, inflating the admin research queue with entries no Researcher can ever meaningfully resolve. Verified on real hardware: version-pinned WinAppRuntime sub-packages match `isSystemComponent()`'s existing `'runtime'` name pattern and land in `systemComponents` (excluded from the queue, as designed), but PowerToys' individual context-menu shims (`Microsoft.PowerToys.FileLocksmithContextMenu` and siblings) match no pattern and land in `unknown`, hitting the queue like any real unrecognized app. Not mitigated this pass — a hardcoded name-pattern list would fix it but conflicts with this project's established "principles over hardcoded exclusions" preference; flagging honestly as a known, small, ongoing admin-queue noise cost rather than silently accepting it or improvising a fix.
- `get_arp_apps()`'s former HKCU gap (per-user installs, e.g. Chrome PWAs, invisible regardless of assessment level) is also fixed — see the ARP bullet in Data sources for apps, above.

## Scanner Stabilization milestone
Small, low-risk pre-signing hardening pass across the whole scanner (`main.rs`) and its immediate server-side support, based on two architecture reviews (discovery quality/noise reduction/metadata/future-proofing generally, then a device-specific follow-up). Deliberately scoped to *improving observation quality*, not expanding discovery scope or touching scoring/recommendations — both reviews concluded the scanner's existing discovery categories were already the right ones; the value left was in identifying what's already found more precisely.

- **Payload/schema versioning**: `ScanPayload` gained `payload_version` (currently `1` — bump only on an actual breaking shape change, not every additive field) and `scanner_version` (`env!("CARGO_PKG_VERSION")`). Added *before* more scanner generations ship without either, since already-distributed exes can never be updated and the server previously had no way to distinguish "old scanner, known shape" from "malformed payload." Passed through into the report object in `scan.js` (`scanner_version`/`payload_version`, `null` for pre-this-change exes) — not interpreted anywhere yet, just preserved for future debugging.
- **`discovery_source`** (`"winget"` | `"arp"`, later also `"running-process"` | `"appx"` — see Observation Enrichment milestone below) added to `AppEntry`, previously computed implicitly by every collector and then discarded.
- **Publisher propagation fix**: winget-sourced apps previously always sent `publisher: None` (`winget export`'s JSON has no Publisher field), which meant `isSystemComponent()`'s Microsoft-gate (below) could only ever fire for ARP-sourced apps — most Microsoft noise slipping past the small Rust-side `skip_prefixes` list (e.g. `Microsoft.EdgeWebView2Runtime`) had no second line of defense. Fixed by deriving a coarse publisher hint from the winget package ID's vendor segment (already computed for name-derivation, zero new cost) — verified end-to-end with a synthetic `Microsoft.SomeRuntime`-style ID reaching `systemComponents` correctly.
- **Publisher merge-order bug, found while fixing the above — don't reintroduce:** `classifyApps()` merges matched apps as `{...entry, ...app}` (DB row, then scanner observation). Once winget apps started carrying a publisher hint, that spread order would have silently overwritten a *real* DB catalog publisher with the scanner's coarse guess for every matched app. Fixed via `mergeEntryWithApp()` (`scan.js`), which keeps `entry.publisher` when present and only falls back to `app.publisher` otherwise — verified against `Microsoft.PowerToys` (DB: "Microsoft Corporation", scanner hint: "Microsoft") surviving intact.
- **Device evidence quality**: `DeviceEntry` gained `manufacturer` and `driver_service` (free on the base `Get-PnpDevice` object, populated in both Quick and Standard/Full) and `driver_provider` (requires `Get-PnpDeviceProperty`, so Standard/Full only — matches the existing arrival/removal-date cost tier; Quick stays presence-only by design). `manufacturer` is used immediately: the "Check →" driver-search link (`report.html`) prepends it when known, for materially better search results on generic device names (e.g. "Focusrite Scarlett 2i2 USB ARM64 driver" instead of a bare device name).
- **MEDIA-class "Likely Native" heuristic corrected** — see the updated bullet in Device/driver detection above. **Known, accepted UX change**: MEDIA-class devices scanned in Quick mode always show "Check" now, including genuine plain USB mics that would previously have shown "Likely native" — a deliberate trade favoring accuracy over optimism, per explicit instruction to "favor conservative conclusions over optimistic assumptions."
- **`is_network` clarified, not renamed**: only ever populated by `get_printers()` (network vs. local printer, via port-name check) — always `false` for `get_devices()` rows, since a Net-class USB-Ethernet/WiFi adapter is already unambiguously labeled via `class`. Documented via comments rather than renamed: a wire-field rename would silently drop the printer "(network)" annotation for any already-distributed scanner exe still sending the old shape, since old exes can never be updated.
- **PowerShell boilerplate deduplicated**: `run_powershell_json()`/`json_as_items()` extracted in `main.rs`, used by every collector (`get_devices`, `get_printers`, `get_arp_apps`, and all four Observation Enrichment collectors below) — previously each collector hand-rolled its own run/parse/error-handling block.

## Observation Enrichment milestone (Guidance Signals)
Originally declared the final planned scanner enhancement before code signing — superseded by one further pass, the Scanner Enrichment milestone (below); see the Distribution section for the current, reaffirmed freeze declaration. Adds relevance signals for a *future* Workspace to weigh — **none of it changes readiness scoring, confidence, or recommendation synthesis today**, and that boundary is structural, not conventional (see below).

**What "Guidance Signal" means, as a standing principle for this codebase:** corroborating relevance evidence for a future recommendation layer to interpret, never a compatibility or recommendation determination made by the scanner itself. The scanner's job stays "observe and report facts"; interpreting what those facts mean for a purchase decision stays server/Workspace-side — the same separation of concerns `isSystemComponent()`/`apps.type` already established for compatibility classification (see System component classification section below), now extended to relevance signals rather than compatibility verdicts.

**New `AppEntry` fields** (apply to every app from every source, existing and new):
- `is_running` — process observed with a visible window at scan time (`Get-Process` + `MainWindowTitle` presence check).
- `is_startup` — name fuzzy-matches a Run-key value name (`HKCU`/`HKLM`/`HKLM\...\Wow6432Node` `...\CurrentVersion\Run`) — **value names only**, never the command strings those values hold (full file paths).
- `has_start_menu_entry` — name fuzzy-matches a `.lnk` base filename under either Start Menu Programs folder (all-users + current-user). No shortcut-target resolution — purely "does a launch entry exist with roughly this name."
- All three use the same fuzzy normalize+contains matching already established for ARP merging (`apply_signals()` in `main.rs`) — same known limitation (imprecise for apps whose registry/shortcut naming differs a lot from their display name), same tolerance level already accepted elsewhere in this codebase. A fourth signal, `is_pinned_taskbar`, was added later in the Scanner Enrichment milestone (below) using the identical pattern — see there for the full writeup.

**New payload arrays, deliberately unscored:**
- `unlisted_apps` — apps discovered via a visible-window running-process scan (`get_running_gui_apps()`) that don't match anything already in `apps` (same fuzzy containment dedup as ARP merging). This is the mechanism for discovering portable/sideloaded software (no installer, no ARP/winget registration) — real-hardware testing found genuine examples this way (a running browser, a game, a chat app) that winget/ARP couldn't see.
- `appx_apps` — AppX/Store packages (`get_appx_apps()`, `Get-AppxPackage`), filtered using the API's own authoritative flags — `IsFramework`/`IsResourcePackage`/`IsBundle`, and `SignatureKind != 'System'` — **not a hardcoded name-pattern list**. `SignatureKind` was added after real-hardware testing showed the first three flags alone left ~141 packages, dominated by deep OS-shell plumbing (`CredDialogHost`, `ShellExperienceHost`, `PinningConfirmationDialog`, and dozens like them) all carrying `SignatureKind = 'System'`; excluding that value cut it to 97 while verified safe. **Don't also exclude `SignatureKind = 'None'`** — that's where PWA-installed web apps (Plex, Home Assistant, etc. — confirmed on real data) show up; excluding it would silently drop genuine user software. Known, accepted residual noise: some component sub-packages (PowerToys' individual context-menu shims, version-pinned WinAppRuntime packages) don't carry any of these flags and still get through — not solved with a name-pattern list, per this project's established "principles over hardcoded exclusions" preference.
- Both arrays were passed through `scan.js` **without ever touching `classifyApps()`** at the time of this milestone — `report = { ...classified, ..., unlisted_apps: req.body.unlisted_apps || [], appx_apps: req.body.appx_apps || [] }`. This guaranteed they couldn't affect `score` or the classified buckets *by construction* (they were never in scope of the function that computes those), not by every future call site remembering not to include them. `report.html` was not touched in this milestone at all. **Superseded for `appx_apps` in the Scanner Enrichment milestone (below)** — `appx_apps` is now deliberately merged into `classifyApps()`'s population (see Assessment levels, above) and no longer passed through as a separate report field. `unlisted_apps` is unaffected and still described accurately by this bullet.

**Privacy design, applied identically across all four new collectors** — the explicit brief was "identify applications, not behavioral information":
- Never capture full file paths (can reveal personal folder/username structure) — only vendor/OS-authored labels (process `Description`/`Company`, registry value names, shortcut filenames, AppX manifest metadata), the same privacy tier ARP/winget already send.
- `MainWindowTitle` is used **only** as a PowerShell-side filter (`Where-Object { $_.MainWindowTitle }`) inside `get_running_gui_apps()` — never selected into the object returned to Rust, so window-title content (document names, browser tab titles, conversation previews) never leaves the machine.
- `get_running_gui_apps()` self-excludes its own process name (`ngpcx-scanner`) and dedupes by process name (one exe can own several windows, e.g. a browser's tabs).

**Verification note**: real-hardware runs before and after this milestone produced byte-for-byte identical `report.html` output (same score, same "Workable with Caveats, Limited Data" text, same app counts) — confirms the new arrays and fields have zero effect on the existing report, not just in theory.

**Deferred / consciously parked observation ideas** (preserve the reasoning, don't re-litigate from scratch):
- **Scheduled Tasks** (logon/boot-triggered, via the Task Scheduler API) — a real, Windows-maintained, comparable sibling to the Run-key startup signal above, and genuinely still open as a small future addition. Not implemented this milestone: unlike Run keys, there's no single authoritative flag (comparable to AppX's `IsFramework`) to separate a 3rd-party logon task from the many built-in Microsoft maintenance tasks Windows ships — would need per-task trigger-type inspection plus new noise filtering, which didn't clear the "lowest-risk, lowest-overhead" bar this milestone was held to.
- **Rejected, with reasons, so they aren't re-proposed:** `Win32_Product` (WMI) — well-documented Microsoft anti-pattern, triggers a full MSI reconfigure/repair as a side effect of querying it; ARP already covers the same ground without that hazard. Event Log install/uninstall history — crosses from "what's present" into "what did the user do and when," conflicting with this milestone's own behavioral-data boundary. File-type/protocol-handler associations — preference data about defaults, not software inventory, weak signal for a purchase decision.
- **Dock aggregation, Thunderbolt/PCIe enumeration, gaming/anti-cheat discovery** — carried forward from the device-focused review that preceded this milestone, still deferred, still for the same reasons: no unified-product identification is possible from VID/PID alone (a dock's sub-interfaces have unrelated vendor VID/PIDs); PCIe devices (true Thunderbolt controllers, eGPUs) have no USB VID/PID to filter on at all; anti-cheat compatibility is a per-game software problem needing a different discovery source (game library manifests) and a separately-maintained dataset, not a device-scanner concern.
- **ARM Optimization Scanner** (per-app PE-header architecture inspection, ARM-only) — see "Future direction: ARM Optimization Workspace" section near the end of this file; still blocked on real ARM64 hardware access, unchanged by either scanner milestone.
- **Device-side and Guidance-Signal score integration** — devices, and now Guidance Signals generally, still contribute nothing to the readiness score, by design. Whether/how a future Workspace should weigh any of this is explicitly a Workspace-redesign decision, not a scanner one — don't fold it into `classifyApps()` without that decision being made deliberately first.

## Scanner Enrichment milestone (computing-environment evidence)
One further scanner-side milestone after Observation Enrichment was declared the final planned enhancement before signing (see Distribution section — that declaration is now amended, not retracted: this milestone closes the loop it left open, and the freeze is reaffirmed going forward from here). Framed around a philosophy shift from "discovering installed software" to "gathering evidence about how the environment is actually used": no individual signal is authoritative, multiple independent signals accumulate into stronger evidence, and missing evidence is never interpreted as evidence of absence. Still scoped to observation quality and coverage, not scoring/recommendation redesign — the same discipline every prior scanner milestone held to.

- **HKCU ARP coverage** — see the ARP bullet in Data sources for apps, above, and the superseded-limitation note in Assessment levels, above.
- **AppX friendly names via `Get-StartApps`** — `get_appx_apps()` now runs `Get-StartApps` first to build a PackageFamilyName→human-readable-Name lookup (correlated via the AppID's `!`-delimited PackageFamilyName prefix), and uses it to set each AppX `AppEntry.name` — falling back to the raw, often hash-prefixed `Get-AppxPackage` package `Name` only when no Start Menu launcher exists for that package (the same residual-noise packages already documented as accepted in Observation Enrichment). Verified on real hardware: `5319275A.WhatsAppDesktop_...` → "WhatsApp", `7EE7776C.LinkedInforWindows_...` → "LinkedIn", `OpenAI.ChatGPT-Desktop_...!ChatGPT` → "ChatGPT Classic". AppX `AppEntry.id` is now the stable PackageFamilyName (previously always `None`) — gives Workspace's Personal Context a stable per-app key for AppX apps, the same way winget/ARP apps already have one.
- **Store/AppX apps now participate in classification** — see the Assessment levels section, above, for the full writeup (the `scan.js` merge, the WhatsApp/LinkedIn/ChatGPT Classic verification, and the accepted admin-queue-noise trade-off).
- **`is_pinned_taskbar` signal** — a fourth Guidance Signal (`AppEntry`), reading `.lnk` base filenames from `%AppData%\Microsoft\Internet Explorer\Quick Launch\User Pinned\TaskBar` (`get_taskbar_pinned_names()`, same fuzzy-match pattern and same doc-comment discipline as `get_start_menu_names()`), wired into `apply_signals()` and exposed in `assessment.js` (`isLightPriorityApp()`, `defaultImportanceForApp()` → `"Pinned to your taskbar"`, `'high'` level, alongside `is_running`/`is_startup`). Taskbar pinning is a deliberate user action — arguably stronger importance evidence than `is_running` (which only proves "happened to be open," not "core to my workflow"). The underlying `.lnk`-folder mechanism is an undocumented legacy Windows API, not a stable/guaranteed-future one; the accepted failure mode if Windows ever changes it is an empty list (missing evidence, never treated as evidence of absence — consistent with this milestone's own guiding philosophy), not a crash or a wrong answer. Verified working on real Windows 11 hardware. Considered and excluded: Start-menu ("Pin to Start") tiles — no functional/documented reader exists (`LayoutModification.xml`/`start2.bin`-adjacent files are Microsoft-undocumented binary blobs, unlike the taskbar's plain `.lnk` folder), so this stays an open, unimplemented idea rather than a "rejected" one — worth revisiting if Microsoft ever documents a reader.
- **Startup enrichment investigated, concluded not practically implementable this pass — self-correction, don't reintroduce the earlier (wrong) framing:** a prior investigation had characterized `HKCU:\...\CurrentVersion\BackgroundAccessApplications\<PackageFamilyName>` as evidence that a Store app "starts with Windows." Dumping the full key (~144 entries) and checking specific per-app values (including a known non-startup control app) showed this was **wrong** — the key lists nearly every installed AppX package regardless of actual startup configuration, with no distinguishing per-app enabled/disabled value in almost all cases. Not implemented as a startup signal; `is_startup` (Run-key based) remains Run-key-only and does not attempt to cover Store apps' startup behavior, which — as far as this investigation found — has no reliable per-app registry signal at all.
- **Research-only findings for the remaining investigated signals** (per this milestone's explicit "do not implement unless one is extremely inexpensive and obviously valuable" bar — only taskbar pinning cleared it):
  - **UserAssist** — deferred at the time this bullet was first written, on a characterization that turned out to be more pessimistic than reality: a follow-up research pass found only two GUIDs actually matter for Windows 10/11 (not a sprawling GUID→category table), and that UserAssist's real value isn't overlap with Prefetch/`is_running`/`is_startup` but a genuinely different axis — it specifically excludes non-interactive launches, which those signals can't distinguish. Implemented in the UserAssist Enrichment milestone, below — this bullet is superseded, not retracted, kept here so the corrected framing is traceable against the original (wrong) one.
  - **Start pinning** — see `is_pinned_taskbar` bullet above; deferred specifically on "no documented reader exists," not on value (it would likely be at least as strong a signal as taskbar pinning).
  - **Default file/protocol handlers** — re-reviewed this milestone, prior rejection (see Observation Enrichment's "Rejected" list) reaffirmed: preference data about defaults, not software inventory: a weak, indirect signal for "is this application part of the user's environment."
  - **Recent-document/Jump List presence** — feasible to read (`AutomaticDestinations`/`CustomDestinations` `.ms` files or the Recent Items folder), privacy-preserving only if scoped to "does a jump list exist for this app" and never its contents (target document names/paths). Deferred: real but modest value over existing signals, and the privacy framing needs more care than this milestone's time budget allowed to get right on the first pass — better to defer than ship a privacy boundary that wasn't carefully reasoned through.
  - **User-installed persistent services** (`Get-Service`, filtered to non-Microsoft `StartType = Automatic`) — cheap and stable, judged moderate-not-obvious value (a background service is weaker importance evidence than a pinned/running/startup app — it's evidence software is *installed*, which is already known, not evidence it *matters*). Explicitly declined despite clearing the cost bar, since the milestone's brief called for at most one addition and taskbar pinning was the stronger candidate.
  - **Notification-area (system tray) presence** — no clean enumeration API exists; the common approaches (parsing the tray's undocumented registry/shell state) are fragile and version-sensitive. Deferred on feasibility, not value.
- **`payload_version` was not bumped** for any of this milestone's changes — all additive fields (`is_pinned_taskbar`, richer AppX `name`/`id`), no breaking shape change, consistent with the existing versioning convention (Scanner Stabilization milestone, above).

## UserAssist Enrichment milestone
One further scanner-side pass, following directly from the Scanner Enrichment milestone's own "Research-only findings" list (above), where UserAssist was flagged as a candidate worth a deeper look rather than a rejected one. A dedicated research phase (Windows registry forensics literature, real byte-offset verification against this machine's own UserAssist data) preceded implementation and corrected an earlier, more pessimistic characterization — see the superseded bullet above.

**What UserAssist is, for this codebase's purposes:** Windows Explorer's own internal launch tracker (undocumented by Microsoft, but stable since Vista). The one thing that makes it worth adding alongside every existing Guidance Signal: real testing (both third-party forensic research and a direct verification pass against this machine's own registry data) confirmed it specifically **excludes non-interactive launches** — Startup-folder and Scheduled-Task launches reliably register a zero run count even though the program genuinely ran. That makes a nonzero UserAssist run count the one signal in this codebase that corroborates a *deliberate, Explorer-driven* launch (double-click, Start Menu, taskbar, search), distinct from every other signal here, all of which prove presence, configuration, or momentary state rather than sustained voluntary use.

**Scanner implementation** (`get_userassist_data()`, `main.rs`): reads two registry GUIDs under `HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist\{GUID}\Count` — `CEBFF5CD-ACE2-4F4F-9178-9926F41749EA` (direct `.exe` launches) and `F4E57C4B-2036-45F0-A9AB-443BCFE33D9F` (shortcut/`.lnk` launches, including AppX/UWP apps, which appear as AUMIDs). Value names are ROT13-decoded (a documented, trivial reversal, not real obfuscation) purely to recover a program name or PackageFamilyName. Run count (byte offset 4) and last-run FILETIME (byte offset 60) are read from the binary value; **entries with a zero run count are discarded entirely**, not treated as weak evidence — this is the mechanism that filters out non-interactive launches. Focus Count/Focus Time are ignored completely: undocumented calculation, inconsistently incremented per the research, and the closest thing to real behavioral profiling among anything considered for this milestone.

**Raw observations, not scanner-generated bands — a deliberate, explicit choice:** `AppEntry` gained `launch_count: Option<u32>` and `days_since_last_launch: Option<u32>`, both un-interpreted numbers, not `"frequent"`/`"occasional"`/`"stale"` labels. This follows the same scanner/assessment split already established for compatibility classification (`isSystemComponent()`/`apps.type`) and Guidance Signals generally: **the scanner is responsible for gathering factual observations; the assessment is responsible for interpreting them.** Reducing to a band inside the scanner would bake a threshold choice (what counts as "frequent"?) into something that can't change without a new signed release; keeping the raw number lets `assessment.js` retune its interpretation freely. `None` on both fields means no UserAssist evidence was found — missing evidence, never evidence of zero use (consistent with this codebase's standing Guidance Signal philosophy).

**Exact timestamps never leave the scanner process, let alone the machine.** The last-run FILETIME is converted to a relative day count *inside the PowerShell script itself* and discarded immediately after — Rust never touches a raw timestamp for this signal, and only the relative age is ever sent to the server. This is a stricter reading than "don't transmit it over the network" — the exact value doesn't survive past the PowerShell subprocess that read it.

**Matching**: `apply_signals()` (extended, not a new function, matching the existing pattern for every prior Guidance Signal) prefers an exact PackageFamilyName match against `AppEntry.id` for AppX apps (unambiguous, since AppX ids are already stable PackageFamilyNames — see Scanner Enrichment milestone) before falling back to the same fuzzy normalized-basename containment check every other signal here already uses. Multiple raw UserAssist records can legitimately match one app (a direct-exe entry and a shortcut entry both tracking the same program) — counts sum, and the smallest (most recent) days-since-last-launch wins, the same "most informative evidence wins" fallback already used for device recency (`IsPresent` OR `LastRemovalDate` OR `LastArrivalDate`).

**Assessment integration** (`assessment.js`) — purely additive, the weighting model (`computeSynthesis()`) is untouched:
- `hasFrequentUserAssistLaunches(app)` (`launch_count >= 3`) and `hasRecentUserAssistLaunch(app)` (`days_since_last_launch <= 30`) are the only two places the raw numbers are interpreted — small, named, easily retuned constants, not a config system.
- Both feed `isLightPriorityApp()` (alongside `is_running`/`is_startup`/`recently_used`/`is_pinned_taskbar`) and `defaultImportanceForApp()` (`"Frequently launched"` at `'high'`, `"Used recently"` at `'normal'`, ranked between `is_pinned_taskbar` and `has_start_menu_entry`) — the same always-visible reason-chip mechanism every existing Guidance Signal already uses, no new UI concept.
- **Advanced/Deeper Research Mode presentation** (`workspace.html`, `evidenceDetailHtml()`): a new `userAssistEvidenceLines(app)` helper (`assessment.js`) returns every applicable interpreted phrase at once (an app can genuinely be both "Frequently launched" *and* "Used recently" — the single-winner reason chip can't express that), rendered as a small "Importance Evidence" checklist inside the existing `.deeper-detail` block, gated behind Deeper Research Mode exactly like Source/Confidence/Compatibility Notes already are. **Raw `launch_count`/`days_since_last_launch` numbers are never rendered anywhere, at any assessment level, including Deeper Research Mode** — only these interpreted phrases ever reach the DOM. Verified directly: searched the live-rendered Results and Workspace DOM for the raw field names and confirmed neither appears.

**Export**: originally shipped as "no changes made" (decision-only export, unrelated to this signal) — **superseded by the Environment Integration Evidence milestone, below**, which extends export to a three-layer format precisely so scanner evidence like `launch_count`/`days_since_last_launch` gets preserved, not just Personal Context.

**Real-scan validation.** Verified end-to-end against this machine's real UserAssist data (19 apps with evidence out of 171 total in a Standard scan): a direct raw-registry cross-check (bypassing the scanner, reading the same two GUIDs manually) confirmed the byte-offset parsing, ROT13 decoding, and zero-run-count filtering are all correct. Concrete example of the validation ask ("does UserAssist identify applications the current assessment would otherwise underestimate"): a Visual Studio installation (catalog-unknown) showed `is_running: false`, `is_startup: false`, `is_pinned_taskbar: false`, `has_start_menu_entry: false` — every existing signal gave it `{level: 'low', reason: null}` — but UserAssist showed a real launch the previous day, correctly promoting it to `{level: 'normal', reason: 'Used recently'}`, confirmed rendering in both the Results reason chip and the Workspace "Importance Evidence" checklist. Several already-prioritized apps (a taskbar-pinned, currently-running browser; a running editor with a Start Menu entry) also carried UserAssist evidence that didn't change their outcome — expected and correct, since they were already correctly prioritized by other signals; this is the "signals reinforcing rather than single-handedly deciding" philosophy working as intended, not a case where UserAssist happened to matter.

**Known limitation, found during validation, not fixed:** a direct raw-data cross-check on a real Chrome installation found the browser's true interactive-launch history split across several UserAssist identities that don't all fuzzy-match back to "Google Chrome" — some Chrome-created sub-entries (PWA/extension launcher shims, named like `Chrome._crx_<hash>.UserData.Profile1`) carry genuine nonzero run counts but don't share a normalizable substring with the browser's own name, so they're silently not counted toward it. This under-counts total interactive launches for apps with this kind of multi-identity registration pattern (Chrome and likely other Chromium-based browsers) — a real, honest gap in the same fuzzy-matching tolerance already accepted for every other Guidance Signal in this codebase, not something a stricter matcher would obviously fix without risking false merges elsewhere.

**Empirical observation worth flagging for future tuning, not acted on this pass:** real launch counts on this test machine were low across the board (max observed: 4; most evidenced apps showed exactly 1) even for genuine daily-driver software — plausible explanation is that UserAssist counts discrete launch *events*, not hours of use, and software that's opened once and left running for long stretches (a common pattern for browsers/IDEs/chat apps) accumulates launches slowly. This suggests the `launch_count >= 3` "Frequently launched" threshold may rarely fire in practice, while the `days_since_last_launch <= 30` "Used recently" threshold is carrying most of the practical value observed so far. Deliberately not retuned based on this single machine's data — the raw-value design exists precisely so this threshold can be adjusted later against broader real-world usage, without a scanner change.

### Focus Fields addendum
A same-evening follow-up to the milestone above, adding `focus_count: Option<u32>` and `focus_time_ms: Option<u32>` to `AppEntry`, explicitly labeled **experimental** and never wired into `isLightPriorityApp()`, `defaultImportanceForApp()`, `isLightDeprioritized()`, or `computeSynthesis()` — only into the Importance Evidence stack (below) as a single conservative line ("Focused when open (experimental)", gated on `focus_time_ms >= 60000`, i.e. at least a minute, to filter pure noise without pretending finer precision).

- **Byte offsets verified against real registry data before implementing**, not assumed from research alone: run count at offset 4, focus count at offset 8, focus time (milliseconds) at offset 12, last-run FILETIME at offset 60 — all within the existing `bytes.Length < 68` guard. Confirmed plausible on real apps (a browser actually used today showed ~47 accumulated focus-minutes; a Remote Desktop client opened briefly showed ~3 seconds).
- **Only trusted from the direct-`.exe` GUID (`CEBFF5CD-...`).** Real-data testing found the shortcut-launch GUID (`F4E57C4B-...`) carries near-zero, non-representative values at these same offsets even for apps with a genuine same-day launch recorded on their direct-`.exe` counterpart (e.g. a `.lnk` entry showing `FocusTimeRaw=1` for an app whose real exe-GUID entry showed 254,687ms) — so `apply_signals()` only ever populates focus fields from direct-`.exe`-sourced `UserAssistEvidence` records, never shortcut-sourced ones.
- **A real bookkeeping entry had to be excluded by name.** `UEME_CTLSESSION` (and the `UEME_CTL*` family) is Explorer's own internal session-control value, not an application — confirmed directly against this machine's registry (a ~1.6KB value, vs. 72 bytes for a real app entry, and a nonsense pre-1970 "last run" date when parsed as a normal record). Excluded by a `-like 'UEME_CTL*'` filter before any byte parsing runs. This also retroactively cleans up the original `launch_count`/`days_since_last_launch` collector, which had been silently absorbing this bogus entry without it ever fuzzy-matching a real app (harmless in practice, but not clean data hygiene).

## Environment Integration Evidence milestone (default handlers, evidence stack, export/import)
Same evening as the Focus Fields addendum, in direct response to a request to complete the evidence model before limited hands-on testing. Three parts: default-handler evidence (new scanner signal), a full Importance Evidence stack (assessment/UI), and a three-layer export/import format (client-side only, no new backend persistence).

### Default-Handler Evidence
`get_default_handlers()` (`main.rs`) reads Windows' per-protocol `UserChoice` registry entries and resolves the winning ProgId to a human-readable app identity, via the same `HKCU\Software\Classes\{ProgId}\Application` → `ApplicationName`/`AppUserModelId` convention Windows itself uses. **Deliberately scoped to protocol handlers only** (`http`, `https`, `mailto`) — file-extension `UserChoice` entries (`.pdf`, `.htm`, `.docx`, etc.) were checked against this machine's real, actively-used registry and found **consistently empty**, not merely hard to resolve, so that half of the original idea is reported as an investigated, deferred limitation rather than guessed at or shipped against zero positive examples. `AppEntry` gained `default_handler_count: Option<u32>` (how many of the checked protocols resolved to this app) and `default_handler_categories: Option<Vec<String>>` (a short coarse list — `"browser"`/`"mail"` today — never a raw ProgId or extension).

- **Matching** reuses the exact same exact-id-then-fuzzy-name contract `apply_signals()` already applies to UserAssist: an AppX-backed handler's `AppUserModelId` PackageFamilyName prefix matches `AppEntry.id` exactly; a Win32-backed handler's `ApplicationName` fuzzy-matches by normalized basename.
- **Real bug found and fixed during testing, don't reintroduce:** a Win32 app can set its own `AppUserModelId` for taskbar-grouping/jump-list purposes (Chrome sets a plain `"Chrome"`) that is *not* an AppX PackageFamilyName — it has no `!` delimiter. The original code treated any non-empty `AppUserModelId` as a PackageFamilyName, which misclassified Chrome's handler evidence as package-based with the wrong key (`"Chrome"` instead of a fuzzy-matchable name), silently failing to attach to the Chrome `AppEntry` at all. Fixed by only extracting a PackageFamilyName when the `AppUserModelId` actually contains `!` — the one structural marker that reliably distinguishes a real AppX AUMID from a Win32 app's own custom identifier. Verified after the fix: Chrome correctly shows `default_handler_count: 2, categories: ["browser"]`; Outlook (a real AppX AUMID) correctly shows `count: 1, categories: ["mail"]`.
- **Confidence framing**: Windows doesn't expose whether a `UserChoice` entry reflects a deliberate user pick vs. an installer-set default with equal certainty for every entry, so this is treated as modest, corroborating evidence — `hasMeaningfulDefaultHandler()`/`defaultHandlerEvidenceLines()` (`assessment.js`) feed `isLightPriorityApp()` and `defaultImportanceForApp()` at the same `'normal'` tier as a Start Menu entry, never `'high'`.
- **Browser/PWA relationship, verified rather than special-cased**: per explicit instruction, Chrome's own launch-count aggregation was *not* hand-tuned to chase every PWA/extension execution identity (see the UserAssist "Known limitation" above — still an accepted gap). Instead, the broader relationship was verified directly on real data: the browser itself is independently evidenced as installed/running/pinned/used (existing signals) *and* as the default handler for web links (this signal); installed browser-hosted PWAs (Google Calendar, TripIt, Home Assistant, etc. — see Scanner Enrichment milestone) already classify and render as separate AppX/PWA entries in their own right, not folded into the browser's count. PWA visibility, not a perfect Chrome execution total, was the goal — confirmed working.

### Importance Evidence stack
`importanceEvidenceStack(app, decision)` (`assessment.js`) replaces the narrower `userAssistEvidenceLines()`-only display built earlier the same evening — it now accumulates *every* independently-corroborating fact currently available (running, startup, taskbar-pinned, UserAssist launch/recency, experimental focus, Prefetch recency, Store/PWA, default-handler, Start Menu, explicit "This matters to me"), not just UserAssist's two facts. Rendered as a checklist in Deeper Research Mode / Advanced detail (`evidenceDetailHtml()`, `workspace.html`) — the ordinary Light/Standard experience is unchanged, still driven by `defaultImportanceForApp()`'s single-winner reason chip. **No single line in the stack determines importance by itself** — showing the accumulated stack instead of a number is the explicit point, not an incidental UI choice. As with every prior evidence surface in this codebase, only interpreted phrases are ever rendered — verified directly that no raw field name or number appears anywhere in the rendered DOM, at any assessment level.

**Known, not fixed:** a pre-existing ARP data-quality issue (two registry entries for the same game, `"Last War:Survival Game"` and `"Last War-Survival Game"`, colon vs. hyphen) produces two `AppEntry` rows whose `itemAnchorId()` (built by stripping non-alphanumeric characters) collapse to the identical DOM id, a real duplicate-id found via this session's own verification pass. Predates this milestone — `itemAnchorId()`'s collision behavior and/or ARP's dedup logic would need to change to fix it, out of scope for tonight's evidence-model work.

### Export/Import: three-layer format
The pre-existing `exportAssessment()`/`handleImportFile()` were decision-only (Personal Context: verified status, notes, `critical_to_me`), preserving no scanner evidence at all for any signal. Rebuilt around three explicit layers (`assessment.js`: `buildExportPayload()`, `buildObservationSnapshot()`, `buildAssessmentSnapshot()`, `sanitizeExportPayload()`), under a new `export_format_version` (currently `1`):

1. **`observation_snapshot`** — the scanner's own facts, stripped of every catalog/classification field via an explicit allowlist (`OBSERVATION_APP_FIELDS`/`OBSERVATION_DEVICE_FIELDS`) rather than trusting the classified bucket objects (which also carry `arm_support`/`source`/`confidence`/etc.) not to leak catalog data into what's supposed to be pure observation. Built from the classified buckets client-side (every classified app already carries its original scanner fields — `mergeEntryWithApp()` in `scan.js` spreads the scanner observation *after* the catalog row) rather than needing a new server endpoint.
2. **`personal_context`** — unchanged shape, the pre-existing decision entries.
3. **`assessment_snapshot`** — a frozen historical record (readiness label, score, confidence, primary reasons reduced to `{app_name, app_id, bucket, kind}`, bucket counts, `model_version`, `catalog_snapshot_at`) computed once at export time via the existing `computeSynthesis()`, never recomputed later.

**Import re-classifies against current catalog state — deliberately, not incidentally.** `handleImportFile()` (`index.html`) now posts the preserved `observation_snapshot.apps`/`unlisted_apps`/`devices` straight through the *existing*, unmodified `POST /api/scan` → `classifyApps()` pipeline (no new server code at all), which is exactly what lets "future versions of CheckMyARM reinterpret older assessments using newer recommendation formulas" work for free — the imported apps get scored against whatever the catalog says *today*, not a replayed old verdict. `scan_mode` is preserved from the original export (previously always hardcoded to the `'imported'` sentinel, which would have silently disabled Light-Assessment deprioritization on any reimported Light session) — falls back to the sentinel only for legacy-format files that never captured it.

**Historical comparison, not replacement.** The frozen `assessment_snapshot` is stashed one-shot (`ngpcx_imported_assessment_snapshot_v1`, consumed exactly once outside `renderReport()` since that function can legitimately run more than once per page load) and rendered as a read-only comparison banner (`renderHistoricalComparison()`, `report.html`) showing the at-import numbers next to the freshly recalculated ones — both remain visible, neither overwrites the other.

**Backward compatible by construction**: `sanitizeExportPayload()` recognizes both the new three-layer shape (`export_format_version` present) and the pre-existing bare `{format_version, entries}` decision-only shape — an already-saved old export file still imports correctly, just without an observation/assessment snapshot to restore or compare against.

**Real end-to-end round-trip verified**, not just unit-level: exported a real Standard-assessment session, fed the file back through `handleImportFile()`, and confirmed the re-imported session reproduced identical bucket counts (34/18/0/96) and carried every scanner field intact (including `launch_count`/`focus_time_ms`) through the reclassification pass, with the historical comparison banner correctly showing matching before/after numbers (same-day reimport, no catalog drift).

**Deliberately not done, per explicit scope boundary:** no server-side retention of exported evidence, no encrypted/obscured format (ordinary versioned JSON), no per-Recalculate-trip history — an export is the saved milestone, not a continuous log.

### Product copy: observed vs. influencing vs. not-prioritized
`computeSynthesis()` now also returns `totalAppCount` (every classified app, matching what the scanner reported) and `countedTotal` (previously an internal-only local variable) alongside the existing `deprioritizedCount`. `report.html`'s reduction-summary sentence gained a second clause — shown only when `deprioritizedCount > 0` (i.e., only under Light Assessment with something actually deprioritized; would be pure redundant noise under Standard/Advanced, where nothing is ever deprioritized) — naming the distinction explicitly: apps identified, apps currently showing strong evidence and shaping the recommendation, and apps not currently prioritized but still visible/promotable. Verified rendering correctly on a real Light-mode session ("Of 148 applications identified, 37 show strong evidence... 111 more are not currently prioritized...").

## System component classification
- Server-side in `scan.js` (moved off the exe deliberately, so pattern tweaks don't need a new signed release). `AppEntry` carries a `publisher` field — real registry data for ARP-sourced apps, a coarse ID-derived hint for winget-sourced apps (see Scanner Stabilization milestone above; the server's `mergeEntryWithApp()` never lets this hint overwrite a better catalog value) — the scanner just reports facts, the server decides what they mean. `isSystemComponent()` in `scan.js` does the Microsoft-publisher-gate + name-pattern heuristic; `apps` table entries with `type: 'system'` (set via the admin tool) override the heuristic going forward.
- System components get their own report bucket (separate from Unknown), muted styling, excluded from score calculation, with a disclaimer noting they're not evaluated and don't affect the score.
- **Public terminology is "Unknown," not "Unclassified"** — the internal `arm_support` value was always `'unknown'`; an earlier phase renamed only the public label to "Unclassified," creating a code/copy mismatch. Reverted site-wide (visitor and researcher-facing copy, plus comments/docs describing the vocabulary) after a compatibility-model review flagged the inconsistency. No behavior, schema, or stored-value change — label only.
- **`apps.architectures` is intentionally left `NULL` on insert** (`merge.js`) — it used to be hardcoded to the literal string `'arm64'` on every row regardless of reality, which is misleading, not just unused. Nothing reads this column today; it stays in the schema, reserved for real per-app architecture detection if that's ever built, but no source currently determines that per-app, so writing a fake value was worse than writing nothing.
- **The homepage's "What We Check" card that used to claim per-executable architecture inspection was corrected** — CheckMyARM has never actually parsed executables or determined per-app x86/x64/ARM64 architecture (the scanner's own `architecture` field is the *system's* CPU architecture, not any installed app's). The card now describes the real capability: comparing installed apps against compatibility data from multiple sources.
- **`apps.notes` is presented as "Compatibility Notes,"** not a general-purpose notes field — same column, same `resolve-app` write path, same `requireResearcherOrAdmin` authorization, just relabeled with an editorial rule in the placeholder ("only include what could materially affect a purchase or deployment decision") and promoted into its own visually distinct callout in `report.html` (previously inline with the Source/Confidence metadata line, which buried the most decision-relevant text in the row under the least important formatting).

## Admin tooling (`server/routes/admin.js`, `server/routes/community.js`)
- Auth is per-route, not a single blanket gate (see User roles & Researcher workflow section for why): `requireAdminAuthOrOAuth` (secret **or** `role = 'admin'`) guards user/role management and destructive deletes; `requireResearcherAuthOrOAuth` (secret **or** `role = 'researcher'`/`'admin'` via rank) guards the research-reading/resolving routes (`GET /unknown-apps`, `GET /stale-apps`, `POST /resolve-app`). The legacy shared secret (`ADMIN_SECRET` env var, via `x-admin-secret` header or `?secret=` query param) is a strict superset of both — it's checked first in each combinator — and stays active alongside OAuth; this is a deliberate, long-running migration window, not a temporary state to "finish." The shared-secret path is **not real security** — deliberately low-stakes — and is meant to be retired via `ADMIN_SECRET_ENABLED=false` only after OAuth/sessions/Railway Volume persistence have all been validated in production. Do not remove or upgrade either path without being asked.
- Five admin/research pages: `admin-research.html` (research unknown apps from scratch — `unknown_apps` table, populated by `trackUnknownApp()` in `scan.js`, sorted by count), `admin-revisit.html` (re-verify existing Emulated/Unsupported verdicts that real scans still hit — gated on `apps.times_matched > 0`), and `admin-community-review.html` (review community evidence and propose/confirm catalog changes — see Community Review section) are all Researcher-or-Admin accessible; `admin-dashboard.html` (work-queue counts, Admin-only) and `admin-users.html` (Researcher request review + role management, Admin-only) are Admin-only — see User roles & Researcher workflow section. All five bootstrap the same way: attempt to load via the visitor's session cookie first (transparent — no secret needed if their role already qualifies), falling back to the `x-admin-secret` entry form only on a 401/403. `admin-research.html`/`admin-revisit.html` hide their `Delete` button client-side for non-Admins (the underlying `DELETE` routes are server-enforced Admin-only regardless) — this is convenience only, per the nav-visibility rule below.
- `admin-research.html`/`admin-revisit.html` share the same resolve action (`POST /resolve-app` → `mergeApp()` with `source: 'admin'`, `confidence: 1.0`, always wins future conflicts) and a `Delete` button for clearing queue/DB entries outright (mainly for test-data cleanup, Admin-only) — this delete only clears the `unknown_apps` queue row, it deliberately does **not** delete the `community_submissions` rows that reference it (see Community trust layer section, "Removed" state).
- Community submissions (`community_submissions` table, `POST /api/community/submit`) let end users report what they found for an app that appeared in their own scan session. See the Community trust layer section below for identity, dedup, weighting, and OAuth — this used to be purely session-scoped and anonymous-only; it isn't anymore.
- Philosophy: **no automated bulk research, no paid search API.** Admin/Researcher manually researches (using the same "Check →" search links end users see) and submits verdicts; `github-lookup.js` (see Data sources) is the one automated exception, and it's deliberately low-confidence/conservative rather than authoritative. Don't propose further automating admin research without being asked.

## Community trust layer: anonymous identity + optional OAuth login
Two phases, both built and both live. Phase 1 (anonymous identity + dedup) works standalone with zero external dependencies. Phase 2 (OAuth) is gated behind `OAUTH_ENABLED` — off by default everywhere, but **now enabled in Railway production** (flipped and tested there directly) as well as locally — see "Deploy-readiness safeguard" below.

**Anonymous identity (`server/identity.js`, Phase 1):**
- First community submission from a browser gets a `ngpcx_anon_id` cookie (`crypto.randomUUID()`, HttpOnly, Secure in production, SameSite=Lax, 400-day maxAge — not a round "2 years": Chrome and the current cookie spec hard-cap `Max-Age` at 400 days regardless of what's requested, so 400 days is the actual ceiling, not an arbitrary choice). Contains no personal data.
- `community_submissions` dedups by identity, not scan session: `anonymous_id`/`user_id` columns, `CHECK (anonymous_id IS NULL OR user_id IS NULL)`, plus partial unique indexes `idx_community_anon_app`/`idx_community_user_app` on `(identity, normalized_name)`. A browser or logged-in user has exactly one submission per app — resubmitting `UPDATE`s the existing row (and resets `state` to `'active'` if it had been superseded), it never inserts a duplicate. This replaced the old `UNIQUE(app_name, session_id)` scoping, which had a real integrity gap: the same browser could submit repeatedly across multiple scan sessions.
- `session_id` stays on the table but now means only "proof this app was in the caller's own scan" (the existing anti-abuse check) — it no longer defines identity or uniqueness. Don't conflate the two again.
- Legacy rows from before this migration (both `anonymous_id`/`user_id` NULL) are preserved as-is by the `db.js` schema-rebuild migration (`schema_version` 2) — visible historically, never counted toward consensus (see weighting below).
- Graceful degradation: if cookies are blocked/cleared, submissions still work but aren't recognized as returning — the existing per-IP rate limit (`isRateLimited()`, 20/24h) is the explicit backstop for this, unchanged.

**OAuth login (`server/passportConfig.js`, `server/routes/auth.js`, `server/sqliteSessionStore.js`, Phase 2):**
- Providers: Google, GitHub via Passport (`passport-google-oauth20`, `passport-github2`). Login is always optional — the anonymous workflow above is never blocked or degraded by OAuth being enabled or disabled.
- `users` table: `id, provider, provider_id, display_name, role ('user'|'reviewer'|'admin'), disabled, contact_opt_in, contact_email, created_at`, `UNIQUE(provider, provider_id)`. New accounts always default to `role='user'` — role elevation is **never automatic** anywhere in this codebase (not by contribution count, agreement rate, or any other metric). Only `email` is deliberately *not* requested as an OAuth scope (`profile`/`read:user` only) — provider_id + display name are sufficient for login, and requesting email merely for a future contact channel would be an unnecessary scope. `contact_opt_in`/`contact_email` exist now but are unused — reserved for a future opt-in reviewer-invitation path (see Known open items).
- Sessions: **not** the scan-tracking `sessions` table — a dedicated `auth_sessions` table (`sid`, `session_json`, `expires_at`), backed by a hand-rolled `express-session` `Store` subclass (`SqliteSessionStore`) over the existing `better-sqlite3` connection. Deliberately not `connect-sqlite3` (pulls in a second native SQLite driver, `sqlite3`, alongside `better-sqlite3` — doubles the native-build surface on Railway) or `better-sqlite3-session-store` (single-maintainer npm package, last published 2022). Session cookie name `ngpcx.sid`, `saveUninitialized: false` (so anonymous visitors who never log in never get an `auth_sessions` row), `rolling: true`, 30-day maxAge.
- **Authorization always reflects live database state, never a cached session value.** `passport.serializeUser` stores only `user.id`; `passport.deserializeUser` re-reads the full `users` row from the DB on *every request*. Revoking a role or setting `disabled=1` takes effect on that user's very next request — no logout, no session expiry needed. Verified directly (not just by code review): a live session held across a direct `UPDATE users SET disabled = 1` immediately started returning 403 on the same cookie, no other action taken.
- Authorization middleware (`server/middleware/auth.js`): `requireAuthenticatedUser` and `requireRole(role)` share one internal check (`resolveActiveUser`) so the disabled check can't be forgotten on one but not the other — 401 if not logged in, 403 if `disabled`, otherwise a rank comparison (`{user:0, researcher:1, admin:2}`, `>=`) so `requireRole('researcher')` also admits admins. Centralized here specifically so role-checking logic isn't scattered per-route. Named aliases `requireAdmin`/`requireResearcherOrAdmin` are exported alongside `requireRole` for auditability at call sites (see User roles & Researcher workflow section) — this rank map originally read `reviewer` instead of `researcher` (defined but never wired to any route); renamed when the Researcher role went live rather than leaving two near-synonym unused/used roles side by side.
- OAuth flow CSRF protection (`server/routes/auth.js`): a server-side `oauth_states` table (single-use, 10-minute expiry) alone is **not sufficient** — it stops open redirects and replay, but doesn't prove the browser presenting the callback is the one that started the flow (classic login-CSRF: an attacker completes their own flow, captures the valid callback URL, and hands it to a victim). Fixed by binding `state` to a short-lived `oauth_nonce` cookie set at flow start; the callback rejects outright (403, before any code exchange) if the query `state` doesn't match the cookie. The final redirect target is *never* taken from client input — always reconstructed server-side from the `state`→`return_session_id` mapping stored at flow start, re-validated against the live `sessions` table.
- Anonymous → authenticated migration runs inside the OAuth callback, in a `db.transaction()`: each of the departing anonymous identity's submissions is reassigned (`anonymous_id` → `user_id`) unless the authenticated account already has a submission for that same app, in which case the authenticated row stays canonical and the anonymous row is marked `state = 'superseded'` (never merged, never deleted, never a second row for that user+app — the partial unique index also structurally guarantees this).
- **Weighting/auto-accept (`tryAutoAccept()` in `community.js`)**: auto-accept requires unanimous agreement **and at least 2 distinct authenticated (`user_id`-bearing) contributors** — anonymous submissions can never trigger it alone, at any count. This is a hard structural gate, not a weighted-sum threshold, specifically because a cookie identity is trivially resettable (clearing cookies / switching browsers produces a new `anonymous_id`) and the per-IP rate limit doesn't stop one person manufacturing several across a day. Anonymous submissions still matter: they still count toward unanimity (their disagreement still blocks acceptance) and still surface apps into the admin queue sooner. A separate, purely informational weighted breakdown (anonymous=1, authenticated=3) is shown to admins on the two admin pages — it never auto-resolves anything, admins still decide disagreements manually.
- Every consensus-related query (`tryAutoAccept()`'s breakdown, `admin.js`'s `submissionsByNormalizedName()` and `COMMUNITY_FLAG_EXISTS`) filters to `state = 'active' AND (anonymous_id IS NOT NULL OR user_id IS NOT NULL)` — superseded and legacy identity-less rows are visible everywhere but never counted. Keep this predicate in sync across call sites if it ever changes.
- My Submissions (`public/my-submissions.html`, `GET /api/community/mine`): a private, non-gamified view of the caller's own submissions (works for anonymous cookie-holders too, not login-gated) — explicitly **no leaderboard, no public profile, no reputation score**. Only one state is actually stored (`community_submissions.state`, `'active'|'superseded'`); everything else displayed (Pending/Accepted/Disputed/Removed) is derived at query time by joining against current `apps`/`unknown_apps` state, not tracked as separate stored events — avoids an entire class of "forgot to update state when an admin acted elsewhere" bugs. Each non-Superseded row has an inline **Edit** control (status buttons + notes, Save/Cancel) backed by `POST /api/community/update` — deliberately narrower than `/submit`: it can only update a row that already exists for the caller's identity, never creates one, and skips re-validating against a live scan session (the original anti-abuse proof was already satisfied once, at the original `/submit` time). Superseded rows aren't editable — they belong to an identity a later authenticated submission has already superseded (see Anonymous → authenticated migration below), so editing one would just reactivate a stale claim.
- Admin/ADMIN_SECRET migration path: `ADMIN_SECRET_ENABLED` (default true) gates the shared-secret branch of `requireAdminAuthOrOAuth`; setting it to `'false'` in Railway retires shared-secret admin access, requiring OAuth+`role='admin'` only. A config change, not a code change — reversible in seconds, and like any `process.env` value takes effect on the next process restart/redeploy, not instantaneously. At no point during the migration is OAuth the *only* admin path unless this is deliberately flipped.
- **Bootstrap / manual role elevation — the permanent process, not a temporary workaround:** log in once via Google or GitHub (creates a `users` row at `role='user'` automatically), find it (`SELECT id, provider, provider_id, display_name FROM users;`), then elevate directly with SQL:
  ```sql
  UPDATE users SET role = 'admin' WHERE provider = 'google' AND provider_id = '1234567890';
  ```
  This remains the only way to create the *first* Admin (a chicken-and-egg bootstrap — `admin-users.html`'s role controls require an existing Admin session to reach at all). Once at least one Admin exists, further role changes (`user`/`researcher`/`admin`, any direction, including demotions) go through `admin-users.html` → `POST /api/admin/users/:id/role` instead — see User roles & Researcher workflow section. Disabling an account (`disabled = 1`) is unchanged: still direct SQL only, no admin UI for it exists or is planned right now — `admin-users.html` shows disabled state read-only.
- **Deploy-readiness safeguard:** `OAUTH_ENABLED` (default unset/false) gates whether the session middleware, Passport, and `/auth/*` routes register at all — if false, none of this code path runs, and the server behaves exactly as it did before this feature existed. If `OAUTH_ENABLED=true` and `SESSION_SECRET` is unset, the server refuses to start entirely rather than falling back to a guessable default (unlike `ADMIN_SECRET`'s `|| 'change-me'`, which is fine for a low-stakes shared secret but wrong for something that can forge login sessions). **Now set to `true` in Railway production** — flipped after the volume persistence pre-flight (below) and full local verification against real Google/GitHub providers; confirmed live via `GET /api/auth-status` returning `enabled:true` with both providers configured.
- **Railway Volume — provisioned and mounted at `/data`** (`ngpcx-volume`, attached to the `ngpcx` service, `DATA_DIR=/data` set in Railway). Originally confirmed via `railway volume list`/`railway logs` that no volume existed and the DB resolved to `/app/data/compatibility.db` (Nixpacks' default build WORKDIR — no Dockerfile/railway.toml/nixpacks.toml in this repo, so that was the default, not a deliberate choice); that's the gap this closes.
- `server/db.js` resolves the DB path via a `DATA_DIR` env var (`DATA_DIR/compatibility.db`, defaulting to the existing repo-relative `data/` if unset — local dev and any environment without `DATA_DIR` set are unaffected). This exists so the volume can be mounted at any path without depending on Railway's internal build directory (`/app`) staying stable.
- **Incident, don't reintroduce:** the first deploy to the empty volume hung indefinitely (Node event loop fully blocked — even unrelated requests like `/health` returned 502) partway through `seedFromCache()`'s original per-row auto-committing insert loop, seemingly a SQLite/volume-filesystem locking issue on the very first write to the freshly-mounted path. Fixed in `server/seed.js`: the entire seed runs inside one `db.transaction()`, so a crash/kill/hang partway through leaves **zero** committed rows (SQLite rolls back the whole incomplete transaction on next open) instead of a silent partial state — verified by force-killing the process mid-transaction and confirming a clean 0-row reopen and reseed.
- **`seedFromCache()` replays both bulk cache sources, not just winget.** A forensic pass (triggered by production's post-volume DB showing only 1,506 rows instead of the historical ~8,883) found the original fix only ever replayed `.cache/winget/` — `.cache/worksonwoa/projects.json` (8,772 entries, also committed to git, also a first-class `enabled: true` scraper in `server/scrapers/index.js`) was never wired into the bootstrap path at all. Not lost data — just an asymmetry in the reseed tooling. Fixed by extracting the exact same per-entry logic the live scrapers use into reusable functions — `saveToDb()` exported from `winget.js`, `buildAppEntry()` extracted and exported from `community/worksonwoa.js` — and calling them from `seed.js`, so the seed never has a second, divergent interpretation of either source's data. Confidence/conflict resolution across the two sources still goes through the normal `mergeApp()` semantics (confidence-gated, agreement bonus), same as a live scraper run would produce.
- Completion is tracked by `meta.seed_sources` (a comma-joined list of which sources completed, e.g. `winget,worksonwoa`) plus `meta.seeded_at`, not just row count or a bare boolean marker — specifically so that when a new source gets added to the expected set (as WorksOnWoA was here), an *older* marker written before that source existed correctly reads as incomplete and safely re-triggers a full reseed, rather than silently leaving the new source unseeded forever. Both markers are written **only if every expected source's cache was found and processed in the same run** — partial availability (e.g. one cache file missing) leaves whatever *did* merge committed, but withholds the marker so the next boot retries the whole seed instead of considering a partially-covered database done.
- Known, deliberately deferred (not restored by `seedFromCache()`, decide separately): ~22 winget-sourced rows present in historical production data but whose cache files no longer exist in the current `.cache/winget/` (real packages — `Microsoft.WSL`, `Microsoft.EdgeWebView2Runtime`, etc. — just cache drift across historical scrape runs); 3 `source='admin'` rows from manual curation, one of which (`admin_test_app_xyz`, "Test App XYZ") is leftover test data, not real.
- The volume-survives-a-redeploy pre-flight (write test data, redeploy, confirm it's still there) was completed before flipping `OAUTH_ENABLED=true` in Railway — see the seed-atomicity incident above, which doubled as that verification (the seeded catalog data demonstrably persisted across multiple redeploys). The app does not and should not try to auto-detect ephemeral-vs-persistent storage from inside the Node process; this stays a manual, one-time judgment call.
- Production also needs `app.set('trust proxy', 1)` (already wired in `server.js`, gated on `NODE_ENV === 'production'`) — Railway terminates TLS at its edge, so without this Express can't see the request as secure and `Secure`-flagged cookies silently misbehave.
- **Explicitly deferred, do not build without being asked:** leaderboards, public profiles, contributor reputation scores, any social/gamification feature. (An in-app role-management UI and a Researcher-invitation workflow — both listed here as deferred in an earlier phase — are now built; see User roles & Researcher workflow section below.) The `contact_opt_in`/`contact_email` columns on `users` exist for a future minimal opt-in "I'm open to helping" path (an admin manually queries `WHERE contact_opt_in = 1` and reaches out by hand) — not built, and deliberately not using OAuth profile email for this, to avoid requesting unnecessary scopes.

## User roles & Researcher workflow
Three authorization roles, unchanged from the Community trust layer section's `users.role`: `user` (default — scan, contribute, edit own submissions, volunteer when eligible), `researcher` (trusted volunteer, see below), `admin` (everything Researcher plus user/role management and full admin tooling). Admin stays small and manually granted — no automatic promotion exists anywhere in this codebase, by role, request count, or any other signal.

**Researcher request is workflow, not a role.** A request (`researcher_requests` table: `id, user_id, status, note, requested_at, reviewed_at, reviewed_by, decision_note`) only ever *causes* a role change when an Admin approves it — the table never itself grants access. `status`: `pending` | `approved` | `declined` | `withdrawn`. A partial unique index (`idx_researcher_requests_pending_user`, `WHERE status = 'pending'`) enforces "at most one pending request per user" at the DB level, not just in application logic — makes repeated "volunteer" clicks harmless even under a race. `decision_note` is Admin-internal only, never returned by any user-facing endpoint.

- **Eligibility** (`server/routes/researcher.js`, `GET /api/researcher/status`): the Researcher-volunteer invitation appears on `my-submissions.html` once a `user`-role account has **3 active community submissions** (`ELIGIBILITY_THRESHOLD`, easy to tune). Meeting it only shows the invitation — it never implies qualification or auto-approval. The endpoint returns a single computed `uiState` (`invite | pending | declined | welcome | none`) rather than raw fields, so the eligibility/cooldown branching logic lives in exactly one place, not duplicated client-side.
- **Declined requests get a cooldown, not a permanent block:** `DECLINE_COOLDOWN_MS` = 30 days from `reviewed_at`. Within the window, `my-submissions.html` shows the declined message; after it, `uiState` reverts to `invite`/`none` based on current eligibility, so re-requesting needs no separate "try again" UI — it's just the normal invitation reappearing.
- **Welcome message** is driven by `users.researcher_welcome_seen` (sticky per-account, not per-request or session-based) plus `role === 'researcher'` — shown once, dismissed via `POST /api/researcher/welcome-ack`. Deliberately not a general notification system; this is the one place a "seen" flag was needed and it's a single boolean column.
- **Approval** (`POST /api/admin/researcher-requests/:id/approve`, Admin-only): marks the request `approved` and sets `users.role = 'researcher'` in one `db.transaction()`. **Decline** leaves `role` untouched — a declined request never disables, punishes, or otherwise marks a normal user; they keep contributing exactly as before.

**Authorization helpers** (`server/middleware/auth.js`): `requireAdmin` and `requireResearcherOrAdmin` are named aliases over the existing rank-based `requireRole()` (`requireRole('researcher')` already admits Admins via the rank comparison — no new logic, just an auditable name at each call site). `server/routes/admin.js` applies these **per-route**, not via a blanket `router.use()` — `GET /unknown-apps`, `GET /stale-apps`, `POST /resolve-app` require Researcher-or-Admin; every user-management, role-change, and destructive-delete route requires Admin. This is a deliberate, minimal expansion of Researcher permissions into the *existing* catalog-review pages (least privilege, easy to audit at each route definition) rather than a new Research workspace or a broad admin-check replacement.

**Admin pages** (both Admin-only, dark-theme-consistent with the rest of the site):
- `admin-dashboard.html` — a work-queue view, not analytics: pending Researcher requests, unknown applications, community findings needing attention (disagreement-flagged), stale entries, authenticated users — each count links to where you'd act on it. No metric invented just to fill the page.
- `admin-users.html` — **Researcher Volunteers** (pending requests first, oldest first, with display name/provider/active+superseded submission counts/account age/request date/optional note/an expandable view of their own community submissions via `GET /api/admin/users/:id/submissions`, and Approve/Decline) then **Users** (default sort: active submissions desc, then account age asc, then `id` asc as a fully deterministic final tiebreaker — `created_at` only has second resolution). Role changes use an explicit select + Apply action (never auto-applies on change) and surface the server's `needsConfirmation` response as a native `confirm()` before resending with `confirmSelf: true`.

**Role-change safeguards** (`POST /api/admin/users/:id/role`, all server-enforced — client-side hiding is convenience only): rejects any role outside `user`/`researcher`/`admin`; rejects malformed/non-integer ids; requires `confirmSelf: true` to change your own role (returns `needsConfirmation: true` otherwise); blocks demoting the sole remaining **active** (non-disabled) Admin — a disabled Admin was never "active," so demoting one is unaffected. This last-Admin check is a plain `COUNT(*)` read immediately followed by the `UPDATE`, with no explicit transaction wrapper needed for correctness: `better-sqlite3` calls are synchronous and Node is single-threaded, so no other request's handler can interleave between the two statements within one process. Authorization always re-reads live `req.user` (role + disabled) on every request via the existing Passport `deserializeUser` — a demoted/disabled Admin or Researcher loses access on their very next request, no logout needed, consistent with the rest of this codebase's auth model.

**Privacy boundaries specific to this feature** (see also the Privacy & Data Governance Audit, `PRIVACY_BACKLOG.md`): contribution counts everywhere in Researcher/user-management UI come from `community_submissions` only — nothing here reads or exposes `sessions.raw_apps`/`sessions.results`, so an Admin reviewing a Researcher request can see *what* someone reported, never their full scan inventory. Provider IDs are never displayed (provider name only, e.g. "Google"). Volunteer notes are visible only on Admin-only endpoints. No email is collected or requested anywhere in this workflow (unchanged from the Community trust layer section's existing scope decision).

**Deferred, do not build without being asked:** a dedicated workspace for *direct catalog resolution* beyond reusing `admin-research.html`/`admin-revisit.html` — that narrower scope is still true today; Community Review (below) is a separate concern (reviewing community evidence and recommendations, not a replacement for direct resolution), and its own dedicated page was a deliberate later decision, not a reversal of this one. Also still deferred: account disable/enable via the admin UI (stays direct-SQL, unchanged from before this phase); reviewer-tier/senior-Researcher roles; any notification system beyond the in-app welcome/pending/declined states already described.

## Community Review (Phase 1)
A dedicated Researcher/Admin workspace (`admin-community-review.html`, backend in `server/routes/admin.js`) for reviewing community evidence and turning it into catalog changes — the gap left after community submissions, inline editing, and Compatibility Notes all existed but had no review workflow of their own. Preserves three deliberately separate concepts, carried through the whole design:
- **Community Findings** — firsthand `community_submissions` rows. Reviewable by a Researcher, never editable by one; only the original contributor edits their own via My Submissions (`POST /api/community/update`).
- **Researcher Recommendations** — a Researcher's evaluation of that evidence (`researcher_recommendations` table). Purely additive: proposing one never touches `community_submissions` or the catalog.
- **The Canonical Catalog** (`apps` table) — the only thing a *confirmed* recommendation updates, and only via the existing `resolveAppCatalog()` path (see below), never a second write path.

**Review queue** (`GET /api/admin/community-review/queue`, `buildCommunityReviewQueue()` in `admin.js`): any app with at least one active, identity-bearing `community_submissions` row qualifies. Deterministic priority sort — conflict (internal disagreement among submissions, or a submission disagreeing with the current catalog verdict) > still sitting in the `unknown_apps` queue with evidence > carries Compatibility Notes > everything else — ties broken by total submission count desc, then app name asc. `buildCommunityReviewQueue()` is the single source of truth for both the queue endpoint and the `dashboard-counts` summary card, so the two can never drift apart. **Don't reintroduce this bug:** "Unknown catalog entries with evidence" means literally still present in `unknown_apps`, not merely "no `apps` row yet" — those are different states (an app can lack a catalog row without ever having been queued as Unknown by a real scan).

**Evidence detail** (`GET /api/admin/community-review/app/:normalizedName`): catalog verdict/source/confidence/notes, the full findings list, and any recommendations. Findings are stripped to `arm_support`/`notes`/`created_at`/a derived `contributor` label (`"Authenticated Contributor"` or `"Anonymous Contributor"`) only — no `anonymous_id`, `user_id`, `session_id`, IP, or scan inventory, matching the Community trust layer's existing privacy boundary. Recommendations, by contrast, **are** attributed by display name (`created_by`/`reviewed_by` joined to `users.display_name`, falling back to a `*_label` like `'shared-admin-secret'` under the legacy secret path) — that's an internal accountability feature for Researchers/Admins reviewing each other's work, not a public-facing one, consistent with how Researcher requests already show display names to Admins elsewhere.

**Recommendation + confirmation workflow** (`researcher_recommendations`, `status`: `'pending' | 'confirmed' | 'rejected'`): a Researcher proposes (`POST .../app/:normalizedName/recommend` — proposed compatibility, optional Compatibility Notes, optional internal rationale). At most one pending recommendation per app at a time (partial unique index in `db.js`) — proposing a second while one is pending returns 409 rather than piling up. A second Researcher or Admin then acts:
- **Confirm** (`POST .../recommendations/:id/confirm`) — applies the recommendation exactly as proposed.
- **Revise** (`POST .../recommendations/:id/revise`) — the second reviewer adjusts the proposed values and applies them in one step. Deliberately a single-step "confirm with edits," not a propose-again/re-confirm cycle — a full revision-history workflow would be the "workflow engine" this phase was explicitly asked to avoid (see Deferred below).
- **Reject** (`POST .../recommendations/:id/reject`) — `status = 'rejected'`, no catalog change, no `community_submissions` change. Unblocks a fresh recommendation for the same app.
- **Leave Unresolved** (`POST .../recommendations/:id/unresolved`) — a genuine no-decision. `status` intentionally stays `'pending'`; `reviewed_by`/`reviewed_at`/`review_note` still update, doubling as a lightweight "last looked at by" marker without a separate touch-tracking column.

**Self-review is blocked, not just discouraged**: `assertNotSelfReview()` in `admin.js` rejects (403) any Confirm/Revise/Reject/Unresolved where the reviewer's `req.user.id` matches the recommendation's `created_by` — server-enforced, not just hidden client-side (the UI also greys these out and shows an explanatory note, using `id` now included in `GET /api/auth-status`'s `user` object specifically to support this). Only meaningful under OAuth; the shared `ADMIN_SECRET` path has no `req.user` and so no self to compare against — same documented limitation as the rest of that legacy path.

**Only a confirmed recommendation touches the catalog, and it reuses the existing resolve path.** `resolveAppCatalog()` in `admin.js` is the one function that ever writes a verdict to `apps` — both `POST /api/admin/resolve-app` (the pre-existing direct admin-research/admin-revisit flow) and Confirm/Revise now call it, so there is exactly one catalog-writing implementation, not two that could drift. Same authorization (`requireResearcherAuthOrOAuth`) governs every entry point into it.

**Stale-recommendation confirmation guard**: a recommendation's proposed verdict is a snapshot of the Researcher's judgment at propose-time — Confirm/Revise must not silently apply it if community evidence has materially changed since. `researcher_recommendations.evidence_snapshot` stores a small JSON fingerprint at proposal time (`computeEvidenceFingerprint()` in `admin.js`): the sorted set of distinct `arm_support` values among active submissions, whether the community agreed with the catalog verdict, a SHA-256 hash of sorted non-empty note text (detects addition/removal/edit without persisting the notes themselves), and a submission count (informational only). **Not a copy of the evidence** — no contributor identity, no raw note text, no scan data. At Confirm/Revise time, `checkEvidenceFreshness()` recomputes the fingerprint live and compares (`compareEvidenceFingerprints()`): material change is a verdict appearing/disappearing from the distinct-opinions set, catalog/community agreement flipping, or the notes hash changing — a plain increase in an *already-present* status's count (another agreeing submission) is deliberately not material and never blocks. A material change returns 409 with `evidenceChanged: true`, a concise `summary` string, and a `changes` breakdown (new/removed verdicts, whether catalog-agreement or notes changed, submission counts before/after) — the recommendation is left untouched (still `pending`, catalog untouched). The reviewer must resend with `acknowledgeEvidenceChange: true` to proceed; doing so prepends an auto-generated note to `review_note` recording that confirmation happened despite the change, for the same accountability reasons as the rest of this section. The self-review guard is checked first and independently — acknowledging an evidence change never bypasses it. `admin-community-review.html`'s `showEvidenceChangeBanner()` handles the 409 by refreshing the card's evidence in place (via the same `refreshCard()` used elsewhere) *before* offering the acknowledgment button, so the reviewer sees current findings, not stale ones, when deciding. **Known lightweight-by-design gap:** the acknowledgment isn't itself versioned against a specific evidence state — a second change arriving in the brief window between viewing the refreshed evidence and clicking acknowledge wouldn't be caught. Acceptable for a low-traffic internal tool; a stricter design would need to re-check freshness at the moment of acknowledgment too, which was judged unnecessary complexity for this phase.

**Research Logging** (`research_activity_log`, append-only, no UI reads it yet): every catalog-changing action (`direct_resolve`, `recommendation_confirmed`, `recommendation_revised`) and every recommendation-lifecycle event that doesn't change the catalog (`recommendation_created`, `recommendation_rejected`, `recommendation_left_unresolved`) writes one row — app name, actor (`actor_id` + `actor_label` fallback, same pattern as `researcher_recommendations`), action, previous/new `arm_support`, previous/new notes, timestamp. Captured now specifically so a future Research Activity page has real history to show rather than needing a backfill.

**Dashboard**: `admin-dashboard.html` gained an "Applications awaiting review" card (`communityReviewApps` from `dashboard-counts`, reusing `buildCommunityReviewQueue()`) linking to Community Review, with a small secondary "N conflicts" indicator (`communityReviewConflicts`) when nonzero.

**Explicitly deferred (do not build without being asked)** — acknowledged, not started: a dedicated Research Activity page; researcher activity history UI; a peer-review dashboard; recommendation revision-history UI; rollback support; evidence-derived confidence scoring; AI-assisted research or automatic recommendation generation; x64/x86 emulation simplification (still tracked as two separate statuses everywhere); a richer device-review workflow (Community Review is apps-only, matching `community_submissions`'s existing app-only scope); public contributor recognition; researcher metrics; notifications; messaging; approval automation.

## Attention & Relevance milestone (post-deploy tester feedback)
The first limited external testing pass on the Product Polish build (above) surfaced real, concrete feedback — described by the person who commissioned it as "assessment/UI refinements rather than scanner problems," and every fix below stayed on that side of the line: `main.rs` is untouched, `scan.js`'s `classifyApps()`/`isSystemComponent()` are untouched. Three problems, all solved client-side in `assessment.js` plus nav markup in the five informational pages.

**1. Session-aware navigation.** Once an assessment session exists, it's now the effective "home" — before this, `about.html`/`why.html`/`privacy.html`/`how-it-thinks.html`/`my-submissions.html` always pointed their logo and back-link at the plain homepage regardless of an active session, stranding a user who clicked through from Results or Workspace.
- `assessment.js` gained `resolveActiveSession()` (reads `?session=`/`&level=` off the CURRENT page's own URL, validates via `GET /api/session/:id`, treats anything other than `status: 'complete'` — missing, expired, still mid-scan — as no session) and `applySessionAwareNav()` (called once on load by the five informational pages; rewrites `#nav-logo-link`/`#nav-back-link` to point at Results, reveals `#nav-workspace-link`, and propagates the session onto every `[data-info-link]` cross-link in the page body). A missing/invalid session is a silent no-op — the plain-homepage markup already in each page is the default, not a fallback branch that has to be triggered.
- `decorateInfoLinks(sessionId, level)` is the other half — called synchronously by `report.html`/`workspace.html` themselves (which already know their own session id at load, no async check needed) to append `?session=&level=` onto their own outgoing nav links to Why/How CheckMyARM Thinks/About/My Submissions, so the thread never drops on the way out either.
- Every page's logo is now wrapped in an `<a>` for the first time (previously a non-clickable `<div>` everywhere) — a small incidental improvement, not just a vehicle for the session-aware href.
- Verified directly (synthetic session, real browser navigation): a valid `status:'complete'` session correctly rewrites logo/back-link/Workspace-link and propagates through every in-body cross-link; a missing session, a bogus session id (404), and no `?session=` param at all all correctly fall back to the plain homepage default with no console errors.

**2. Attention eligibility — "does this deserve the user's attention."** The concrete complaint: Windows platform/shell AppX packages (`Microsoft.WindowsNotepad`, `Microsoft.VP9VideoExtensions`, `Microsoft.Winget.Source`, `Microsoft.XboxIdentityProvider`, `Microsoft.MicrosoftSolitaireCollection`, PowerToys context-menu shims) were dominating Needs Attention purely for having `is_running` or `has_start_menu_entry` — the two weakest, most passive signals in the whole Guidance Signal set, true of nearly any OS-shell process. Explicit instruction: a small number of durable architectural rules, not a name list. This is a genuine evolution of the standing Guidance Signal philosophy (CLAUDE.md, Observation Enrichment milestone) — **"Evidence should support importance. Evidence alone should not automatically create importance"** is the new governing sentence alongside the old one, not a replacement for it.
- **`isPlatformComponent(app)`** — structural, not a name list: an AppX (`discovery_source === 'appx'`) package-family-name matching `PLATFORM_COMPONENT_ID_PATTERN` — `^Microsoft\.(Windows|Xbox|Gaming|Microsoft)` (Microsoft's own first-party inbox/shell/gaming-integration naming conventions, including the redundant "Microsoft.Microsoft*" doubled prefix real Microsoft Studios inbox apps use — MicrosoftSolitaireCollection, MicrosoftStickyNotes) or a generic plumbing keyword (`Extension(s)`, `Provider`, `.Source`, `ContextMenu`, `Overlay`, `SparseApp`, `Service(s)`). Never applied to winget/ARP apps — the convention is Microsoft-AppX-specific.
- **`hasDeliberateUseEvidence(app)`** — `is_startup || is_pinned_taskbar || hasFrequentUserAssistLaunches || hasRecentUserAssistLaunch || hasMeaningfulDefaultHandler` — every signal that reflects an actual human choice, deliberately excluding `is_running`/`has_start_menu_entry`.
- **`qualifiesForAttention(app, decision)`** — the eligibility gate itself: `critical_to_me` always passes; deliberate-use evidence always passes; a platform component with only passive evidence fails; everything else (an ordinary winget/ARP/AppX app with `is_running` or `has_start_menu_entry`, still the majority case) passes exactly as before. This is a PRIOR question to `defaultImportanceForApp()`, not a replacement for it — passing items are still ranked/labeled by the existing single-reason-chip logic unchanged.
- **`classifyAssessmentItem()`** gained a 4th outcome, `lowSignal`, checked only for genuinely Unknown apps (`bucket !== null`) that fail the gate — settled/verified/known-bucket items are unaffected, and devices never go through this gate at all (unchanged: "a Light-scanned device list is already just 'currently connected'"). Callers (`report.html`'s `renderReductionSummary`, `workspace.html`'s `bucketItems()`) now pass `app` through.
- **New UI surface, both pages**: `report.html` gained a `renderLowSignal()` collapsed section (same `preview-list`/`<details>` pattern as Not Currently Prioritized, deliberately a separate section — orthogonal axis, an item can be in both/either/neither, same precedent as the existing Not-Currently-Prioritized-vs-Needs-Attention relationship). `workspace.html` gained a `lowSignal` group in `bucketItems()`/`renderGroups()`, same collapsed-by-default treatment as Already Considered/Set Aside. Low-signal items are still fully classified, scored, visible, and promotable ("This matters to me" always overrides) — this only changes where they're grouped for attention, never the compatibility bucket or the score.
- **Real fix found during validation, not anticipated up front:** `report.html`'s reduction-narrative loop (`renderReductionSummary`) never checked `isLightDeprioritized()` before this pass — a pre-existing gap (present before this milestone too) that let a Light-deprioritized app appear in the "Needs Attention" preview AND in the separate "Not Currently Prioritized" section simultaneously. Fixed by skipping Light-deprioritized apps at the top of the loop, matching what `workspace.html`'s `bucketItems()` already correctly did. Verified via a synthetic Light-mode scan: before the fix, a deprioritized app appeared in both places; after, it correctly appears only in Not Currently Prioritized.
- **Verified end-to-end against a synthetic Standard scan** (real HTTP session + `/api/scan` payload, not just code reading): `Microsoft.VP9VideoExtensions`, `Microsoft.Winget.Source`, `Microsoft.XboxIdentityProvider`, `Microsoft.PowerToys.FileLocksmithContextMenu`, and `Microsoft.MicrosoftSolitaireCollection` all correctly routed to Low-Signal Items instead of Needs Attention; Google Chrome, Microsoft Teams, Visual Studio Code, and WhatsApp (none AppX-platform-pattern matches, or not AppX at all) kept full normal evidence treatment and landed in Needs Attention/Already Considered exactly as before — confirming Teams/365/VS-style real applications are structurally exempt from this gate, not coincidentally spared.

**3. Browser-hosted PWA compatibility inheritance.** A Chrome/Edge/Firefox-installed PWA (`messages.google.com`, TripIt, and similar HKCU-ARP-discovered entries — see Scanner Enrichment milestone) has no independent ARM64 binary of its own; it runs entirely inside its host browser's rendering engine, so an independent "? Unknown" verdict for it is architecturally misleading, not merely unresolved.
- **`isBrowserHostedPWA(app)`** — `discovery_source === 'arp'` and `publisher` matches a known browser-vendor prefix (`Google\Chrome`, `Microsoft\Edge`, `Mozilla\Firefox`, `BraveSoftware\Brave-Browser`) — the exact per-user ARP Publisher string the scanner already reports for these entries (Scanner Enrichment milestone); no new scanner signal needed.
- **`findHostBrowserEntry(app, allApps)`** locates the host browser's own entry within the SAME scan population by normalized name, and **`effectiveBucketFor(app, bucket, decision, allApps)`** is the one shared priority function — personal verification first (an explicit human claim), then PWA inheritance for a still-Unknown app (an architectural fact), then the raw catalog bucket — used identically by `computeSynthesis()` (score/confidence), `classifyAssessmentItem()` call sites (attention grouping), and `pwaInheritanceNote()` (UI transparency), so scoring and grouping can never silently disagree about an app's effective bucket the way two independent inline copies of this logic previously risked.
- **Never a silent override**: the PWA's own row still renders in its raw "? Unknown" bucket/badge everywhere (`report.html`'s app table, `workspace.html`'s status badge) — same "row stays where it renders" precedent as the existing `personally_verified` badge — with an ADDITIVE note ("Runs inside Google Chrome — treated as Native ARM64 for your Score, since it has no separate ARM64 binary of its own") shown next to it, always visible, not gated behind Deeper Research Mode.
- Considered and deliberately NOT done: `Angry Birds Friends`, named alongside `messages.google.com`/TripIt in the original feedback as a "PWA-like" example, turned out on inspection to be a genuine Store/AppX game package (`discovery_source: 'appx'`, not an ARP entry with a browser-vendor Publisher) — structurally NOT a browser-hosted PWA by this milestone's own detection logic, so it correctly does NOT inherit anything and stays an independent Unknown verdict. Flagging this rather than forcing a special case for it, per the same "principled model over individual exceptions" instruction the rest of this milestone followed.
- Verified end-to-end against a synthetic scan: two ARP entries with `publisher: "Google\\Chrome"` alongside a real "Google Chrome" winget entry (native) both correctly inherited Native, contributed to score/confidence as Native, landed in Already Considered (not Needs Attention/Low-Signal), and rendered the inheritance note on both Results and Workspace.

**Known, not fixed — a pre-existing catalog-data artifact found during validation, unrelated to this milestone's own logic:** a synthetic "Notepad" app (`Microsoft.WindowsNotepad`) fuzzy-matched to an `emulated` catalog entry in local dev data rather than `native` — almost certainly stale/unrelated catalog data (or a coincidental fuzzy-name collision) rather than a bug in this milestone's code, since `isPlatformComponent()`/`qualifiesForAttention()` never touch classification, only attention grouping. Not investigated further or "fixed" by editing catalog data, since altering `apps` table contents wasn't part of this task and the actual production catalog may already have this correct.

## Background / Reference milestone (Report vs Workspace separation)
A direct follow-up to the Attention & Relevance milestone above, after the user's own real assessment still felt like "a filtered software inventory" rather than a recommendation. Core principle, stated explicitly and now load-bearing across this whole area: **the catalog supports the assessment — it does not become the assessment.** Everything the scanner/catalog find stays fully discoverable (Workspace, especially Advanced); not everything needs to be surfaced as if it were an equal decision driver. Still entirely client-side (`assessment.js` + `report.html`/`workspace.html`); `main.rs` and `scan.js` remain untouched.

**1. Unified background/reference classification** — the prior milestone's `isPlatformComponent()` (AppX package-ID keyword patterns) missed real examples from the user's own machine (`Microsoft.PowerAutomateDesktop`, `Microsoft.CommandPalette` — neither matches any plumbing keyword or the `Microsoft.Windows*`/`Xbox*`/`Gaming*`/`Microsoft.Microsoft*` prefixes). Replaced with a broader, simpler, more durable TYPE test:
- **`isMicrosoftAppxPackage(app)`** — `discovery_source === 'appx'` AND (publisher contains "microsoft" OR the package family name `id` starts with `Microsoft.`). Two signals, not one, found necessary during validation: `classifyApps()`'s fuzzy name-matching can pair an AppX package with an unrelated third-party catalog entry, and `mergeEntryWithApp()` lets that catalog row's publisher win over the scanner's own — a synthetic `Microsoft.WindowsNotepad` package matched an unrelated catalog app and inherited its publisher (`"Arsh Digital"`), which would have silently defeated publisher-only detection. `app.id` (the package family name) is scanner-reported and never overwritten by the catalog merge, so it survives this failure mode. Deliberately still publisher/id-based, never a package-ID keyword list — "is this Microsoft's own first-party AppX software," not "does this name look like plumbing."
- **`isBrowserHostedPWA(app)`** — unchanged from the prior milestone (ARP + browser-vendor Publisher).
- **`isBackgroundCandidate(app)`** = `isMicrosoftAppxPackage(app) || isBrowserHostedPWA(app)` — a structural TYPE verdict, independent of which compatibility bucket the app resolves to.
- **`hasBackgroundOverride(app, decision)`** — `critical_to_me`, `personally_verified`, or real deliberate-use evidence (`hasDeliberateUseEvidence`) — the one way a background-typed item earns full treatment. Passive evidence (`is_running`/`has_start_menu_entry`) is deliberately never an override.
- **`isBackgroundApp(app, decision)`** = `isBackgroundCandidate(app) && !hasBackgroundOverride(app, decision)` — the single verdict used identically by attention grouping (`classifyAssessmentItem`), score/confidence weighting (`computeSynthesis`), and Report framing, so they can never silently disagree. The prior milestone's `qualifiesForAttention()` helper became redundant under this design and was removed.
- `classifyAssessmentItem()`'s outcome was renamed `lowSignal` → **`background`**, and its check now runs BEFORE the resolved/unresolved check (previously only genuinely-Unknown apps could be low-signal) — a background-typed app stays background-weighted/grouped even when the catalog does have a real native/emulated/unsupported verdict for it, since the whole point is this kind of software rarely drives a purchase decision either way. Both pages' UI renamed "Low-Signal Items" → **"Background Items"**.

**2. Score/Confidence weighting** — new `BACKGROUND_WEIGHT = 0.25` constant (same order of magnitude as `DOESNT_MATTER_WEIGHT`, a distinct concept, retunable independently). Applied inside `computeSynthesis()`'s per-app loop identically to `doesnt_matter`/`critical_to_me` — a background app's Score/Confidence contribution is multiplied down, REGARDLESS of whether its bucket is resolved or still Unknown (an inherited-native browser-hosted PWA is still down-weighted; "rarely a valuable assessment driver" is a statement about influence, not merely about uncertainty). Background-typed unresolved apps are also excluded from `unresolvedCount` (the "N unresolved apps are holding back a higher confidence rating" sentence) — at 0.25 weight they barely move Confidence, so counting them there would overstate their real impact. `scoreReasons` (Primary Reasons) now excludes background apps outright, not just down-weights them — Primary Reasons is explicitly "why this recommendation," and background software shouldn't headline it even on a large raw point swing.

**3. Report vs Workspace: curate vs expose.** `computeSynthesis()` returns two new counts, `materialCount` (apps that are NOT background — what actually drove today's recommendation) and `backgroundCount`. `report.html`'s reduction-narrative headline was rewritten around them: *"We analyzed N applications. M materially influenced today's recommendation. K are worth a closer look — sharing what you know about them helps you and the next person. Everything else — including Microsoft platform software and browser-hosted apps — stays fully available in the Workspace to review or contribute to."* Replaces the old "We evaluated N applications and devices... X already provide enough information..." framing, which counted every resolved app as if it were an equal contributor. The four raw app-bucket tables (`buildTable()` — native/emulated/unsupported/unknown) are now collapsed-by-default `<details>` for every type, not just System Components (a small unconditional-`<details>` generalization, dropping the old type-specific branch) — the full inventory stays exactly as discoverable (one click), it's just not presented as if every row were an equal decision driver.

**4. Community contribution alignment.** The refresh banner's copy (`renderRefreshBanner()`) now names the actual small "worth a closer look" count (the same population `renderReductionSummary()` already computes, cached in `currentNeedsAttentionCount`) — *"3 applications worth your attention don't have a confirmed verdict yet..."* — instead of a generic "some apps don't have a confirmed verdict yet," which used to fire for any of dozens of Unknown/Emulated/Unsupported rows regardless of relevance. Falls back to a background-specific message when nothing material is unresolved, rather than staying silent or misleadingly generic.

**Verified end-to-end against a realistic synthetic Standard scan** (15 apps: Chrome/Teams/VS Code/WhatsApp as real user software; the user's own named examples — WindowsNotepad, VP9VideoExtensions, Winget.Source, XboxIdentityProvider, MicrosoftSolitaireCollection, PowerToys.FileLocksmithContextMenu, PowerAutomateDesktop, CommandPalette — as Microsoft AppX; two Chrome-hosted PWAs plus one Edge-hosted PWA where Edge itself isn't in the scanned population, to exercise the uninheritable case):
- Headline correctly read "We analyzed 15 applications. 4 materially influenced today's recommendation" (vs. a naive "15 analyzed, 7 already resolved" framing the old copy would have produced).
- All 8 named Microsoft-platform examples correctly routed to Background Items regardless of resolved bucket — including `Command Palette`, which fuzzy-matched a real catalog `native` entry and still correctly stayed background-weighted rather than becoming a Primary Reason.
- The Edge-hosted PWA with no Edge in the population correctly showed a plain "? Unknown" badge with no fabricated inheritance claim — the two Chrome-hosted PWAs correctly inherited Chrome's Native status with the existing transparency note.
- `critical_to_me` ("This matters to me") correctly overrides the background verdict, confirmed directly against `classifyAssessmentItem()`.
- Results and Workspace agreed exactly on both counts (4 material / 11 background) with no console errors.
- **Known test-environment artifact, not a product bug:** the animated score gauge (`animateScore()`, driven by `requestAnimationFrame`) rendered "0" in this headless browser-automation pane while the underlying `computeSynthesis().score` was correct and the readiness label/explanation text rendered correctly — `requestAnimationFrame` appears not to tick reliably in this specific automated context. `animateScore()` itself was not touched by this milestone.
- **Limitation, stated plainly:** this environment has no real Windows machine or physical scanner available, so "use a real assessment while refining the model" was necessarily approximated with a realistic synthetic population built from the user's own named real-world examples, not their actual machine's scan. The user should re-run a real Light/Standard assessment to confirm the before/after populations match this validation.

## Canonical Effective Status milestone (consistency correction)
A focused correction after real-machine testing of the Background/Reference milestone found several places still displaying or grouping by the RAW catalog bucket while the assessment had already computed a different EFFECTIVE status for the same item — e.g. a browser-hosted PWA whose badge said "? Unknown" right next to a note saying "treated as Native ARM64." Root cause: `effectiveBucketFor()`/`isBackgroundApp()` were being called independently at several UI call sites (Report tables, Report counts, Workspace badges, the reduction narrative) instead of from one shared source, so they could silently drift apart. Still entirely client-side (`assessment.js` + `report.html`/`workspace.html`); `main.rs` and `scan.js` remain untouched.

**1. `classifyPopulation(results, personalContext)`** (`assessment.js`) — the one canonical per-app classification: `{app, rawBucket, effectiveBucket, isBackground, decision}`. `computeSynthesis()` now builds its working set from this instead of re-deriving `effectiveBucketFor()`/`isBackgroundApp()` inline in three separate places (the main loop, `scoreReasons`, `confidenceReasons`) — a real bug this surfaced: `confidenceReasons` was checking the RAW bucket for "still unresolved," so a critical-to-me PWA that had successfully inherited Native from its browser could still wrongly show as "limiting confidence." `report.html`'s `renderReport()` now computes `classified` once and threads it through counts, tables, and `renderReductionSummary()`.

**2. Badges always show the EFFECTIVE status, never raw.** `buildTable()` (`report.html`) and `statusBadge()` (`workspace.html`) both read `effectiveBucket`, not the raw catalog bucket — a browser-hosted PWA that inherited Native from Chrome now shows "✓ Native ARM64 (inherited)" as its primary badge, not "? Unknown" with a separate explanatory note that used to read as contradictory. Raw catalog status, when it differs from effective, is now shown ONLY in the expandable detail row (Report) / Deeper Research Mode (Workspace), explicitly labeled "Raw catalog status:" — never blended with the primary badge. `checkLink()` also switched to effective bucket, so an inherited-Native PWA no longer offers a "Check →" research link for something already resolved.

**3. Background apps excluded from the primary Report tables entirely, not just re-labeled.** Previously a background-typed app (Microsoft AppX platform software, browser-hosted PWA) still appeared in the raw Native/Emulated/Unsupported/Unknown tables AND in the separate Background Items section — genuinely duplicated. `materialByBucket` (`report.html`) now excludes both `isBackground` AND deprioritized apps before populating the four primary tables and their stat-count cards, so those counts always equal `synthesis.materialCount` broken out by bucket — no more "Unknown (7)" in the table while the headline says "2 unresolved apps." System Components reuses the same `{app, rawBucket, effectiveBucket, decision}` shape (`rawBucket === effectiveBucket === 'system'`) for consistency, even though it has no real bucket concept.

**4. Background status is checked BEFORE deprioritization, not after — a real ordering bug.** `isDeprioritized()` (Light/Standard footprint gate) used to run first in `computeSynthesis()`, `bucketItems()` (`workspace.html`), and `renderReductionSummary()` (`report.html`), which meant a background-typed app with no footprint evidence (true of most platform AppX packages) got silently absorbed into "Not Currently Prioritized" instead of "Background Items" — and `backgroundCount` swung between assessment levels even though background-ness is a structural, level-independent fact. Fixed by checking `isBackground` first everywhere: a background app now always lands in Background Items and always counts toward `backgroundCount`, regardless of scan level; deprioritization only ever applies to genuinely non-background apps. Verified directly: `backgroundCount` is now identical (11) across a Light and a Standard scan of the same synthetic population, where before the fix it read 5 under Light and 11 under Standard for no principled reason.

**5. Standard's material population is no longer "everything not background."** `isDeprioritized()` gained a real Standard-tier gate — `isStandardPriorityApp(app)` (`has_start_menu_entry || isLightPriorityApp(app)`) — broader than Light's strict gate (which deliberately excludes `has_start_menu_entry`) but still a real bar, not "any catalog match counts." Previously *every* non-excluded, non-background app counted toward `materialCount` under Standard regardless of evidence, which is why a real Standard scan reported "92 materially influenced" — essentially the full non-background catalog population. Verified against a realistic synthetic 22-app population: Light `materialCount` 5, Standard `materialCount` 6 (one footprint-only app, `has_start_menu_entry` but otherwise untouched, correctly promotes from deprioritized under Light to material under Standard) — a small, real, credible difference, not "Standard = Light with a lower Confidence bar."

**6. Terminology: "Needs Attention" → "Deserves Attention"**, throughout `assessment.js` (`classifyAssessmentItem()`'s return value), `report.html`, and `workspace.html` — code identifiers (`groups.needsAttention` → `groups.deservesAttention`, `needsAttention` array/count variables) renamed alongside the display text, not just the label, so the two can't drift back apart later. `currentNeedsAttentionCount` → `currentDeservesAttentionCount`.

**Real bug found and fixed while doing this rename, unrelated to the rename itself:** `workspace.html`'s wiring loop (`['notPrioritized', 'needsAttention', 'alreadyConsidered', 'setAside'].forEach(...)`, which calls `wireItemCard()` to attach decision-button/checkbox listeners) never included `'background'` — added in the prior milestone but never wired into this loop, so every card in Background Items rendered with dead buttons. Fixed by adding `'background'` to the array.

**Verified end-to-end against the same realistic synthetic 22-app population used in the prior milestone**, extended with 5 zero-footprint "Ghost Utility" apps (no evidence of any kind — the exact shape of the real-world noise that produced "92 materially influenced") and one footprint-only app to exercise the Light/Standard boundary:
- Standard: "We analyzed 22 applications. 6 materially influenced today's recommendation" (down from what would have been ~17 under the pre-fix "everything not background" definition) — all counts reconcile exactly (4 Native + 0 Emulated + 0 Unsupported + 2 Unknown = 6 material; 6 + 11 background + 5 not-currently-prioritized = 22 total).
- Light: 5 materially influenced, 6 not currently prioritized, 11 background (same background count as Standard) — confirms Light and Standard now have meaningfully different material populations from the identical underlying scan, not just a Confidence-tier difference.
- Browser-hosted PWAs (Chrome-hosted, inheritable) showed "✓ Native ARM64 (inherited)" consistently on both Results and Workspace; an Edge-hosted PWA with no Edge in the population honestly showed "? Unknown" with no fabricated inheritance claim, on both pages, in both the primary badge and the Background Items list.
- No item appeared in two groups simultaneously (spot-checked "Not Currently Prioritized" vs. the primary Unknown table — zero overlap after the fix, five items of overlap before it).
- No console errors, no duplicate DOM ids, confirmed via direct DOM inspection across both pages and both assessment levels.
- **Known test-environment artifact, unrelated to this milestone:** the animated score gauge (`animateScore()`, `requestAnimationFrame`-driven) again read "0" in this headless browser-automation pane while `computeSynthesis().score` itself was verified correct via direct inspection — the same pre-existing environment limitation documented in the prior milestone; `animateScore()` was not touched.

## Browser-Hosted PWA Compatibility milestone (real-machine follow-up)
A focused follow-up after real-machine testing of the Canonical Effective Status milestone found one remaining materially misleading case: browser-hosted PWAs (Chrome/Edge/Firefox/Brave-installed, ARP-discovered — see `isBrowserHostedPWA()`) still showed as `? Unknown`. The prior design ("inherit the classified bucket of whichever specific browser entry is present in this scan's own population," via `findHostBrowserEntry()`) had two real flaws the user's testing surfaced: it silently reverted to Unknown whenever that specific browser wasn't itself present in the scan population, and it implied — wrongly — that the future ARM PC must run that exact browser to get the benefit. Still entirely client-side (`assessment.js` + `report.html`/`workspace.html`); `main.rs` and `scan.js` remain untouched.

**The governing product rule, now load-bearing:** browser-hosted applications inherit compatibility from the browser platform and are presented as supported through a compatible ARM64 browser — not as Unknown merely because they have no independent ARM64 binary. This is a TYPE fact about the app (it has no ARM64 binary to classify, by architecture, not by data gap), never contingent on which specific browser happens to be in this scan's population — Chrome, Edge, Firefox, and Brave all ship real ARM64-native builds today, so the buying-decision conclusion holds regardless of which one the user currently uses.

**1. New `'supported'` effective bucket** — a bucket value distinct from native/emulated/unsupported/unknown, assigned unconditionally by `effectiveBucketFor()` whenever `bucket === 'unknown' && isBrowserHostedPWA(app)` (no host-browser lookup, no dependency on any specific browser's presence or bucket). `findHostBrowserEntry()`/`hostBrowserNameFor()`/`BROWSER_HOST_APP_NAMES` were deleted outright, not deprecated — the new design has no use for "which browser hosts this" at all, only "is this confidently a browser-hosted PWA." Badge label: "✓ Browser-Hosted (Supported)," a third distinct color (`.badge.supported`, teal `#2dd4bf` in `assessment.css`) — not native's green (it isn't a literal ARM64 binary verdict) and not check's blue (nothing here needs further research). `pwaInheritanceNote()` was renamed `pwaSupportedNote()` and rewritten to drop all browser-specific wording: *"This browser-hosted application does not require a separate ARM64 binary. It can be installed or used through a compatible ARM64 browser on your ARM PC."* No "(inherited)" badge suffix anymore — the `supported` bucket value is already self-explanatory, so the badge/inherited-tag machinery in both `buildTable()` (`report.html`) and `statusBadge()` (`workspace.html`) was removed, not adapted.

**2. Raw catalog status stays visible, never contradicts.** The existing "Raw catalog status: Unknown" line (Report's expandable detail row, Workspace's Deeper Research Mode) is unchanged — it still fires whenever `rawBucket !== effectiveBucket`. What changed is only the primary badge/effective bucket it's shown alongside.

**3. Scoring: `'supported'` contributes like a resolved positive fact, tracked separately from `nativeCount`.** `computeSynthesis()`'s points formula is now `(effectiveBucket === 'native' || effectiveBucket === 'supported') ? 100 : ...` — same 100 points as native, since it's a genuinely resolved, positive compatibility fact, not "not applicable." A new `supportedCount` variable (returned from `computeSynthesis()`, used in `scoreReasons` too) is tracked deliberately separate from `nativeCount` so the explanation sentence can never claim a browser-hosted app "runs natively" — it gets its own clause instead: *"N more are browser-hosted and supported through a compatible ARM64 browser."* This is the sentence-level enforcement of "don't imply the user must use the same browser" / "don't describe it as currently running inside an ARM64 browser," not just a badge-level one.

**4. Compatibility treatment and importance treatment remain fully independent — unchanged from the prior milestone, verified still true.** `isBrowserHostedPWA(app)` alone makes an app a background *candidate* (`isBackgroundCandidate()`), same as before; `hasBackgroundOverride()` (critical_to_me, personally_verified, or real deliberate-use evidence — running/pinned/frequent/recent-UserAssist/default-handler) is still the one escape hatch, untouched by this milestone. A pinned or actively-used PWA (e.g. LinkedIn) can still land in material apps and even headline a Primary Reason with its honest "Browser-Hosted (Supported)" badge; a passively-detected PWA with no override evidence still recedes into Background Items. `isBrowserHostedPWA()`, `isBackgroundCandidate()`, `hasBackgroundOverride()`, `isBackgroundApp()`, `classifyAssessmentItem()` were not modified by this milestone — only how a `'supported'`-effective-bucket PWA is *labeled and scored* changed, never whether it's material.

**5. Bug found and fixed while wiring this up:** `report.html`'s `materialByBucket` object (populates the four primary compatibility tables from the canonical `classified` population) was initialized as `{ native: [], emulated: [], unsupported: [], unknown: [] }` — no `supported` key. Since a background-overridden PWA (e.g. a pinned LinkedIn) now has `effectiveBucket === 'supported'` and correctly reaches this material path, pushing into `materialByBucket['supported']` would have thrown (`Cannot read properties of undefined`). Fixed by adding `supported: []` to the initializer and a new `#supported-section` table (same `buildTable()` call pattern as the other four, same collapsed-by-default treatment) between Native and Emulated in the Report's app-inventory list.

**6. System Components de-duplicated from Background Items on ordinary Reports.** Separate real-machine finding: Microsoft software was correctly landing in Background Items (client-side, AppX-based `isMicrosoftAppxPackage()`) but *also* still rendering in the server-side System Components section (`results.systemComponents`, an entirely independent mechanism — see System component classification section) on the same Light/Standard Report, reading as redundant even though the two mechanisms don't literally duplicate the same row (System Components apps are removed from the native/emulated/unsupported/unknown populations server-side before they'd ever reach `classifyPopulation()`). Fixed by gating both the `#system-section` table render and its explanatory note (`system-component-note`) to `results.scan_mode === 'full'` only — Light and Standard Reports no longer show System Components at all; Advanced (full) Report is unchanged, still the place "full categorization" lives. Workspace was never a consumer of System Components at all (confirmed via grep — no `systemComponents` reference anywhere in `workspace.html`), so this gating removes nothing from Workspace, consistent with "don't remove data from the Workspace."

**Verified**: `node -c` on both `assessment.js` and the extracted inline `<script>` blocks of `report.html`/`workspace.html` passed clean. Full grep sweep confirmed zero remaining references to `findHostBrowserEntry`, `hostBrowserNameFor`, `BROWSER_HOST_APP_NAMES`, or the old `pwaInheritanceNote` name anywhere in `public/`.

## Future direction: ARM Optimization Workspace (parked — hard blocker on hardware access, do not design/implement without being asked)

**The idea:** when the scanner detects it's running ON Windows on ARM hardware (not just checking readiness to migrate TO one), the workspace reframes its question from "should I buy an ARM PC?" to "how well-optimized is my ARM PC?" Same infrastructure (workspace, community, confidence model) — the recommendation logic and framing shift, not the architecture. Part of a broader lifecycle: Research → Purchase → Optimize → Contribute → Help the next buyer.

**The one genuinely new capability this requires:** per-app executable architecture detection (real PE-header inspection to determine if a specific installed binary is actually ARM64-native, x64, or x86) — the scanner does not have this today. Confirmed via the compatibility-model codebase audit: there's no PE-header parsing anywhere in `main.rs`, and `apps.architectures` is dead/hardcoded data, not real per-app detection. Don't assume this is "just exposing data already collected" — it isn't, it's new engineering (likely the `goblin` or `pelite` crate, a new `Cargo.toml` dependency).

**Correctly scoped (after a few rounds of narrowing — this is the resolved scope, not the first draft):**
- **ARM-only.** Per-app architecture inspection only makes sense to run when the scanner has confirmed it's on real ARM64 hardware — nothing on an x86/x64 machine can ever be ARM64-native, so there's nothing to observe there. This isn't a cut corner, it's the feature's own logic drawing its natural boundary.
- **Target set: everything NOT already confirmed Native by the database** — not "unresolved apps only" (an earlier, too-narrow draft). This single filter covers both real use cases at once: (a) apps marked Emulated → surfaces genuine upgrade opportunities to the user ("you're running the emulated build, a native one exists"), and (b) apps marked Unclassified → improves the database (reality may have moved past the last scrape). No separate list-loading step needed — this is just `classifyApps()`'s existing output, partitioned into "confirmed Native" vs. everything else.
- **Observed evidence corroborates, never determines compatibility.** An app observed running natively on ARM64 proves it *launches*, not that it's *fully functional* — the Logi Tune case (runs via Prism, but RightSight/Vibe 100 features are unsupported) is the concrete example. Scanner-observed evidence should raise confidence in an existing determination, not outrank a qualified Researcher or community finding, and should never be treated as equivalent to "fully verified."

**Hard blockers, not design questions — don't attempt to solve these on paper:**
1. No ARM64 hardware available to build or test against. Any PE-parsing logic designed without real hardware to verify it risks repeating the `is_arm`/`PROCESSOR_ARCHITEW6432` mistake — a fix that shipped without ever being confirmed against the hardware it targets, because none was available at the time.
2. Depends on `is_arm` detection (the `PROCESSOR_ARCHITEW6432` fix) being correct — it was applied but has never been verified against real ARM64 hardware either. Verify that first, independently, before this feature's trigger condition can be trusted.

**When to revisit:** once real ARM64/Snapdragon hardware is available for testing — not before. Until then, this stays a documented idea, not a prompt.

**Expected implementation impact:** likely scanner-only plus workspace presentation; no expected changes to the trust model or community workflow.
