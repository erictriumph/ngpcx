# NGPCX — ARM Readiness Scanner

## What this is
Free Windows ARM (Snapdragon) readiness scanner. Consumer-facing brand: CheckMyARM — live on the visitor-facing pages (see Branding & monetization section); redirect domain still pending. Learning project with real utility; affiliate monetization is a bonus, not the goal.

## Stack — do not deviate without asking
- Server: Node.js + Express, **plain JavaScript, no TypeScript**. SQLite (better-sqlite3). Runs on Railway, auto-deploys from GitHub on push.
- Scanner: Rust .exe (`scanner/src/main.rs`), runs locally on the user's Windows machine only.
- No React, no Next.js, no Tailwind. No running the scanner from the web server. No running scrapers on Windows in production.

## Architecture essentials
- Scan level (quick/standard/full) travels through the localhost:7878 handshake between browser and scanner — never CLI-flag-only, never separate exe builds. CLI flags exist for dev convenience and always win over the browser-provided level (see `get_cli_scan_mode()` precedence in `main.rs`).
- `--local` flag shortens the 7878 timeout to 3s (vs 20s prod) for faster dev iteration.
- Session IDs use `crypto.randomUUID()` (Node). Sessions expire after 24h — this is enforced server-side in both `/api/session/:id` and the scan-submission handler in `scan.js`, not just written and ignored.
- All HTML rendering of app/device names goes through an `escapeHtml()` helper (defined separately in each of `report.html`, `admin-research.html`, `admin-revisit.html` — no shared JS file) — never insert external data into `innerHTML` unescaped. This project has a real stored-XSS history (device FriendlyName is attacker-influenceable via USB descriptors); do not reintroduce raw interpolation. `escapeHtml()` encodes `"`/`'` in addition to `&`/`<`/`>` — needed because app/device names get interpolated into HTML *attributes* (`data-name="..."`, etc.) in the admin pages, not just text nodes, and a name containing a quote character could otherwise break out of the attribute. Found and fixed this gap once already; don't reintroduce a version that only handles text-node context.
- Readiness score (`classifyApps()` in `scan.js`) counts Unclassified apps as **presumed emulated** (same 60-point weight as confirmed Emulated), not excluded — Windows on ARM runs ordinary x86/x64 apps under emulation by default, so true incompatibility is the rare case, not "no data either way." The separate confidence indicator (`report.html`, "High/Moderate/Limited Confidence — N of M apps matched") deliberately stays scoped to native+emulated+unsupported only, excluding Unclassified — it answers "how much do we actually know," a different question from the score's "how likely is this to work." Don't collapse these two back into one number.

## Data sources for apps (in `server/scrapers/`)
- `winget.js` — GitHub-search-based, ARM64 installer detection, confidence 0.95
- `community/worksonwoa.js` — WorksOnWoA DB, confidence 0.75–0.90. Only maps `compatibility: 'no'` → `unsupported`; genuinely captures negative data when the source has it, but the source itself is ~99.9% "yes" (checked the raw feed: 6 "no" entries out of 8,772) — it's a "here's what works" community database, not a "here's what's broken" one. Don't assume there's untapped negative data being discarded here; there isn't.
- `github-lookup.js` — on-demand, single-app GitHub repo search + latest-release asset check (arm64/aarch64/win-arm64 in filenames), confidence 0.70. Triggered fire-and-forget from `scan.js` *after* the scan response is sent, for Unclassified apps (cooldown via `unknown_apps.last_lookup_attempt`) and Emulated/Unsupported apps (cooldown via `apps.last_updated`) — never blocks the triggering user's report. Requires a strict normalized-name match on repo name (not `full_name`) plus a minimum star count (`MIN_STARS`, currently 20) before accepting a match — a wrong match is worse than no match. Confidence is deliberately kept below WorksOnWoA's floor (0.75) so it can never silently override a real source, only fill gaps or reinforce agreement with another github-auto entry.
- ARP (`get_arp_apps()` in `main.rs`) — Windows Uninstall registry, supplements winget for apps it misses (Office, Adobe, vendor tools). No confidence scoring; feeds `merge_arp_into_apps()` which dedupes against winget-found apps by normalized name — mode-aware, see App tiering section.
- **Never build a scraper against Windows Update Catalog or Microsoft Store's undocumented catalog API** — both investigated and rejected: Catalog requires POST+viewstate (a plain GET returns a static shell) and conflicts with Microsoft's ToU on reproducing catalog content; Store's catalog API is undocumented with unclear ToS. If asked to revisit either, flag the prior investigation before proceeding.

## Report refresh + on-demand lookups
- The classification loop lives in `classifyApps(apps)` (`scan.js`), extracted specifically so it can be re-run later against current DB state without a new physical scan. `POST /scan` calls it once at submission time; `sessions.raw_apps` persists the exact submitted app list so `POST /api/session/:id/refresh` (`server.js`) can call `classifyApps()` again later and pick up anything an admin resolved, a community submission accepted, or `github-lookup.js` found in the meantime.
- `report.html` shows a "Refresh results" banner whenever anything's still Unclassified/Emulated/Unsupported, with copy that's deliberately honest about the mechanism: the automated check already ran once (fire-and-forget, right after the original scan), not an ongoing background process — repeatedly clicking refresh with no change is expected, not broken, since nothing new happens until the 14-day lookup cooldown elapses or a human (admin/community) acts.
- **Idempotency gotcha, don't reintroduce:** `renderReport()` can run more than once per page load (initial poll, then again on every refresh click). Any note it inserts into the DOM (`quick-mode-note`, `standard-mode-note`, `system-component-note`) must be removed before being re-added (`document.getElementById(id)?.remove()`), or it duplicates on every refresh — this was a real bug found via actual use, not just review.

## Device/driver detection (`main.rs`)
- Classes scanned: Printer, Image, MEDIA, Biometric, SmartCardReader, Camera, HIDClass, **Net, Display**. The last two were added specifically to catch USB-Ethernet/WiFi adapters and DisplayLink-style USB docks/graphics dongles — historically the worst ARM Windows compatibility pain point (Surface Pro X dock issues) — previously invisible to the scanner entirely. Filtered to `VID_xxxx&PID_xxxx` pattern InstanceIds (excludes `SW\` virtual bus devices); this same filter is the primary defense against virtual/software network adapters (VPN TAP, Hyper-V vswitches) showing up as Net-class noise, since those don't carry real USB VID/PID identifiers.
- Composite devices (one physical device, multiple USB interfaces) are deduped by VID/PID. Display name prefers non-generic interface names; "fido" is explicitly preserved even though it matches the generic-name filter (security keys must not display as "Keyboard Generic"). Classification uses the **most conservative class across all interfaces** — a device with any Biometric/SmartCardReader interface stays flagged for review even if it also exposes a generic HID interface.
- Recency signal is `IsPresent` OR `LastRemovalDate` OR `LastArrivalDate` (in that fallback order) — never arrival date alone. A continuously-connected device never updates arrival date; a recently-unplugged device is better signaled by removal date than by a stale arrival date. Tiers: Quick = present only (no per-device property lookups — this is what makes Quick fast); Full = everything. **Standard's window is per-class**, via `standard_threshold_days()`: 14 days for Camera/HIDClass/MEDIA/Biometric/SmartCardReader (these tend to cluster into "near-daily" or "effectively abandoned" — printers already bypass this filter entirely), 60 days for Net/Display (a travel dock or backup-drive adapter used monthly is a real pattern, closer to how apps are tiered than to a webcam).
- **Gotcha, don't reintroduce:** `Get-PnpDevice -PresentOnly:$false` means "show ONLY non-present devices," not "ignore presence." Omit the parameter entirely for true unfiltered enumeration.
- Printers via `Get-Printer` (not `Get-PnpDevice` — network printers have no USB VID/PID at all). Deduped by `DriverName`. Always included regardless of scan tier (no reliable recency signal exists for them).
- "Likely native" driver-check heuristic: device class is Camera, HIDClass, or MEDIA (standard USB device classes with Windows-inbox drivers — UVC, HID, USB Audio Class) — validated against real hardware (webcam, USB mic, BT-dongle-as-USB-audio, HID input). Stays "worth a check" (search link, no claim) for Biometric/SmartCardReader and vendor-specific printer drivers — validated as genuinely ambiguous via real search results (Kensington fingerprint reader case).
- Driver/app "Check" links point to a plain web search (`"<name> ARM64 driver"` or similar) — **not** Windows Update Catalog (see rejection above).

## App tiering (`main.rs`)
- Two independent recency signals feed `include_by_recency(mode, is_running, days_ago)`: Prefetch `.pf` file mtime (`get_recent_app_names()` — presence alone isn't enough, the file's *last-modified* time is what tracks actual usage recency) and currently-running processes (`get_running_process_names()`, via `Get-Process`) — the latter exists specifically because Prefetch can be disabled system-wide (common on SSD installs), which Prefetch-only detection can't see at all. Quick = any Prefetch record or currently running; Standard = used within 60 days, no recency data at all (mirrors the device tiering "unknown = keep" rule — a missing Prefetch record doesn't prove staleness), or currently running; Full = everything.
- `merge_arp_into_apps()` is mode-aware — it applies the exact same `include_by_recency()` check per ARP-sourced app (matched by name against the same Prefetch/running signals) before merging. **Don't reintroduce the old bug**: this function used to take no mode parameter at all and merge the entire ARP catalog into every tier including Quick, silently defeating Quick's "fast, recent-only" promise for any app winget didn't already find.
- Apps stay at a flat 60-day Standard threshold (not per-class like devices) — a once-a-month app (finance/tax software, etc.) is a normal pattern with no good way to sub-categorize apps the way device *class* sub-categorizes hardware, so the window stays generous.

## System component classification
- Server-side in `scan.js` (moved off the exe deliberately, so pattern tweaks don't need a new signed release). `AppEntry` carries a `publisher` field (threaded through `merge_arp_into_apps()`); the scanner just reports facts, the server decides what they mean. `isSystemComponent()` in `scan.js` does the Microsoft-publisher-gate + name-pattern heuristic; `apps` table entries with `type: 'system'` (set via the admin tool) override the heuristic going forward.
- System components get their own report bucket (separate from Unclassified), muted styling, excluded from score calculation, with a disclaimer noting they're not evaluated and don't affect the score.

## Admin tooling (`server/routes/admin.js`, `server/routes/community.js`)
- Auth: shared secret via `ADMIN_SECRET` env var, checked via `x-admin-secret` header or `?secret=` query param. **Not real security** — deliberately low-stakes, meant to be manually removed from the public deploy when not in active use. Do not upgrade this without being asked; the low-effort approach was a deliberate choice, not an oversight.
- Two admin pages, two distinct workflows: `admin-research.html` (research unknown apps from scratch — `unknown_apps` table, populated by `trackUnknownApp()` in `scan.js`, sorted by count) and `admin-revisit.html` (re-verify existing Emulated/Unsupported verdicts that real scans still hit — gated on `apps.times_matched > 0` so it only surfaces apps someone's machine actually reported, not the full bulk-scraped catalog). Both share the same resolve action (`POST /resolve-app` → `mergeApp()` with `source: 'admin'`, `confidence: 1.0`, always wins future conflicts) and a `Delete` button for clearing queue/DB entries outright (mainly for test-data cleanup).
- Community submissions (`community_submissions` table, `POST /api/community/submit`) let end users report what they found for an app that appeared in their own scan session — the session tie-in is the anti-abuse mechanism (can't submit for an app you didn't scan), plus a per-IP rate limit as a fallback. Five unanimous submissions auto-accept (confidence 0.8) but *only* for apps with no existing `apps` row; any disagreement, or any submission contradicting an existing verdict, never auto-resolves — it surfaces as a badge/flag on the two admin pages above instead of a third page. A `user_id` column exists on the table, reserved and unused, for a planned future logged-in community with attribution/leaderboard — not built yet.
- Philosophy: **no automated bulk research, no paid search API.** Admin manually researches (using the same "Check →" search links end users see) and submits verdicts; `github-lookup.js` (see Data sources) is the one automated exception, and it's deliberately low-confidence/conservative rather than authoritative. Don't propose further automating admin research without being asked.
